================================================================================
ASP.NET CORE — MIDDLEWARE, FILTERS, AND THE DIFFERENCE BETWEEN THEM
================================================================================

--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- HTTP Request Pipeline ---

Technical definition:
  The HTTP request pipeline is the ordered sequence of components (middleware)
  that process an incoming HTTP request and build the outgoing HTTP response
  in an ASP.NET Core application. Each component in the pipeline receives the
  request, can do work, and either passes it to the next component or stops it.

Simple way:
  When a request comes in (e.g., GET /users), it does not jump straight to your
  controller. It travels through a chain of components one by one. Each
  component can read, modify, log, or block the request before passing it on.
  The response travels back through the same chain in reverse order.

  Request  →  [MW1] → [MW2] → [MW3] → Controller
  Response ←  [MW1] ← [MW2] ← [MW3] ← Controller


--- Short-Circuiting ---

Technical definition:
  Short-circuiting is when a middleware component handles a request and returns
  a response without calling the next component in the pipeline. The request
  stops there and does not continue further.

Simple way:
  If a middleware decides "this request should not go further" (e.g., the user
  is not authenticated), it returns a response immediately — the request never
  reaches your controller.


--- Action Invocation Pipeline ---

Technical definition:
  The action invocation pipeline is a sub-pipeline within the MVC framework.
  It runs only for requests that have been matched to a controller action.
  Filters run inside this pipeline — they are aware of the controller, action,
  arguments, and result in a way that middleware is not.

Simple way:
  Middleware runs for every request, even ones that never reach a controller.
  The action invocation pipeline only starts after routing has matched a request
  to a specific controller action. Filters live inside this inner pipeline.

--------------------------------------------------------------------------------
1. MIDDLEWARE
--------------------------------------------------------------------------------

Technical definition:
  Middleware are software components assembled into the ASP.NET Core request
  pipeline using the IApplicationBuilder interface (or WebApplication in
  minimal APIs) in Program.cs. Each middleware component receives an
  HttpContext object, can perform work before and after the next component,
  and decides whether to call next() to continue the pipeline or return early
  to short-circuit it.

Simple way:
  Middleware is code that runs on every single HTTP request that enters your
  application — before and after it reaches your controller. Each middleware
  component handles one cross-cutting concern (logging, authentication, CORS,
  compression, etc.). They are chained together in order in Program.cs.

--- How Middleware Works ---

  app.Use(async (context, next) =>
  {
      // Code here runs BEFORE the next middleware
      Console.WriteLine("Before");

      await next();   // pass to the next middleware in the pipeline

      // Code here runs AFTER the next middleware returns
      Console.WriteLine("After");
  });

  The request goes "down" the chain (before next()), hits the endpoint or
  terminal middleware, then comes back "up" the chain (after next()).


--- Registering Middleware (Program.cs) ---

  Order in Program.cs is the order middleware runs. This matters.

  var app = builder.Build();

  app.UseExceptionHandler("/error");  // must be first — catches all errors
  app.UseHttpsRedirection();
  app.UseStaticFiles();
  app.UseRouting();
  app.UseCors();
  app.UseAuthentication();            // must come before UseAuthorization
  app.UseAuthorization();
  app.MapControllers();

  app.Run();


--- Built-in Middleware ---

  Middleware                  | What it does
  ---------------------------|----------------------------------------------------
  UseExceptionHandler()      | Catches unhandled exceptions globally
  UseHttpsRedirection()      | Redirects HTTP requests to HTTPS
  UseStaticFiles()           | Serves static files (CSS, JS, images) directly
  UseRouting()               | Matches the request URL to an endpoint
  UseCors()                  | Applies Cross-Origin Resource Sharing rules
  UseAuthentication()        | Identifies who the user is (reads JWT/cookie)
  UseAuthorization()         | Checks if the identified user is allowed access
  UseRateLimiter()           | Limits requests per client per time window
  UseResponseCaching()       | Caches responses and serves them for repeat calls
  UseResponseCompression()   | Compresses response bodies (gzip, brotli)


--- Types of Middleware by Behaviour ---

  Pass-through middleware:
    Does work, then calls next() — lets the request continue.
    Example: logging middleware — logs the request, then passes it on.

  Terminal middleware (short-circuit):
    Does work and returns a response without calling next().
    Example: if UseStaticFiles finds the file, it returns it immediately
    and the request never reaches the controller.

  Branch middleware (Map / MapWhen):
    Creates a separate pipeline branch for requests matching a condition.

    app.Map("/health", branch =>
    {
        branch.Run(async context =>
            await context.Response.WriteAsync("OK"));
    });
    // Requests to /health are handled here and go nowhere else.


--- Custom Middleware ---

  Option 1 — Inline (simple, one-off use):

    app.Use(async (context, next) =>
    {
        // log every request
        Console.WriteLine($"Request: {context.Request.Method} {context.Request.Path}");
        await next();
        Console.WriteLine($"Response: {context.Response.StatusCode}");
    });

  Option 2 — Class-based (preferred for reusable, testable middleware):

    public class RequestLoggingMiddleware
    {
        private readonly RequestDelegate _next;

        public RequestLoggingMiddleware(RequestDelegate next)
        {
            _next = next;
        }

        public async Task InvokeAsync(HttpContext context)
        {
            Console.WriteLine($"Incoming: {context.Request.Path}");
            await _next(context);
            Console.WriteLine($"Outgoing: {context.Response.StatusCode}");
        }
    }

    // Register in Program.cs:
    app.UseMiddleware<RequestLoggingMiddleware>();

  RequestDelegate is the type that represents the next middleware in the chain.
  InvokeAsync is called by the pipeline for each request.

2. FILTERS
--------------------------------------------------------------------------------

Technical definition:
  Filters are components in the ASP.NET Core MVC action invocation pipeline
  that run code at specific, well-defined stages around the execution of a
  controller action. They are implemented as attributes or registered globally
  and have access to MVC-specific context — action arguments, model binding
  results, the action descriptor, and the result object.

Simple way:
  Filters are hooks that run at specific moments around a controller action —
  before it runs, after it runs, when it throws an exception, etc. Unlike
  middleware, they know about controllers, actions, and model binding. They
  are more targeted — you can apply them to a single action, a whole
  controller, or globally.


--- The Filter Pipeline (Order of Execution) ---

  For a single controller action, filters run in this order:

  Request arrives at controller
        ↓
  1. Authorization Filter
        ↓
  2. Resource Filter (OnResourceExecuting)
        ↓
  3. Model Binding
        ↓
  4. Action Filter (OnActionExecuting)
        ↓
  5. Action Method runs
        ↓
  6. Action Filter (OnActionExecuted)
        ↓
  7. Exception Filter (if exception occurred)
        ↓
  8. Result Filter (OnResultExecuting)
        ↓
  9. Action Result executes (JSON written, View rendered)
        ↓
  10. Result Filter (OnResultExecuted)


--- Filter Type 1: Authorization Filter ---

Technical definition:
  Authorization Filters are the first filters to run. They implement
  IAuthorizationFilter or IAsyncAuthorizationFilter. If the check fails,
  they short-circuit the pipeline by setting context.Result, preventing
  all subsequent filters and the action from running. The built-in
  [Authorize] attribute is an authorization filter.

Simple way:
  The bouncer filter — runs first, checks if the user is allowed in.
  If not, it stops everything and returns 401 Unauthorized or 403 Forbidden.
  Your action method never runs.

Example:
  [Authorize]              // built-in authorization filter
  public IActionResult GetOrders() { ... }

  [Authorize(Roles = "Admin")]  // only Admin role allowed
  public IActionResult DeleteUser(int id) { ... }


--- Filter Type 2: Resource Filter ---

Technical definition:
  Resource Filters implement IResourceFilter or IAsyncResourceFilter and run
  before and after the rest of the filter pipeline (except authorization).
  They run before model binding, making them suitable for short-circuiting
  early (e.g., serving from cache) or for handling resource-level concerns.

Simple way:
  Resource filters wrap the entire action execution including model binding.
  Use them when you want to short-circuit very early — for example, returning
  a cached response before model binding and the action even happen.

Example use case:
  A custom cache filter that checks if a cached response exists.
  If yes — return the cached response immediately, skip the action entirely.
  If no — let the action run, then cache the result.


--- Filter Type 3: Action Filter ---

Technical definition:
  Action Filters implement IActionFilter or IAsyncActionFilter. They run
  immediately before (OnActionExecuting) and after (OnActionExecuted) the
  controller action method. At the OnActionExecuting stage, the action
  arguments are already bound from the request. The filter can read, modify,
  or replace arguments, and can short-circuit by setting context.Result.

Simple way:
  Action filters run right before and right after your action method.
  They have access to the action's input arguments and the result it produced.
  This is the most commonly used filter type — for logging, validation,
  modifying inputs, or transforming outputs.

Example — custom action filter:

  public class LogActionFilter : IActionFilter
  {
      public void OnActionExecuting(ActionExecutingContext context)
      {
          // runs BEFORE the action
          Console.WriteLine($"Executing: {context.ActionDescriptor.DisplayName}");
          Console.WriteLine($"Arguments: {string.Join(", ", context.ActionArguments.Keys)}");
      }

      public void OnActionExecuted(ActionExecutedContext context)
      {
          // runs AFTER the action
          Console.WriteLine("Action completed.");
      }
  }

  Applied as an attribute:

  [TypeFilter(typeof(LogActionFilter))]
  public IActionResult GetUser(int id) { ... }


--- Filter Type 4: Exception Filter ---

Technical definition:
  Exception Filters implement IExceptionFilter or IAsyncExceptionFilter.
  They run when an unhandled exception is thrown by the action method or
  by an action filter. They can handle the exception by setting
  context.Result (which marks the exception as handled) or they can let
  it propagate. They do NOT catch exceptions from resource filters,
  result filters, or middleware.

Simple way:
  Exception filters are the safety net for controller actions. If your
  action throws an exception and nothing else caught it, the exception
  filter gets the chance to handle it — log it, return a friendly error
  response, etc.

  Important limit: they only catch exceptions from actions and action
  filters. For everything else (middleware errors, result filter errors),
  you still need middleware-level error handling.

Example:

  public class ApiExceptionFilter : IExceptionFilter
  {
      public void OnException(ExceptionContext context)
      {
          context.Result = new ObjectResult(new {
              error = context.Exception.Message
          })
          {
              StatusCode = 500
          };

          context.ExceptionHandled = true;  // marks exception as handled
      }
  }


--- Filter Type 5: Result Filter ---

Technical definition:
  Result Filters implement IResultFilter or IAsyncResultFilter. They run
  before (OnResultExecuting) and after (OnResultExecuted) the action result
  executes — meaning after the JSON is serialised or the view is rendered.
  They only run if the action completed without an exception.

Simple way:
  Result filters wrap the final step — writing the response. They run just
  before the JSON or HTML is written to the response, and just after it is
  written. Use them to add response headers, modify the response format, or
  log after the response is sent.

Example use case:
  Adding a custom response header to every successful action result:

  public void OnResultExecuting(ResultExecutingContext context)
  {
      context.HttpContext.Response.Headers.Add("X-Api-Version", "1.0");
  }


--- How to Apply Filters ---

  1. As an attribute on a single action:
     [TypeFilter(typeof(LogActionFilter))]
     public IActionResult GetUser(int id) { ... }

  2. As an attribute on a whole controller (applies to all actions):
     [TypeFilter(typeof(LogActionFilter))]
     public class UserController : ControllerBase { ... }

  3. Globally (applies to every action in the app) — in Program.cs:
     builder.Services.AddControllers(options =>
     {
         options.Filters.Add<LogActionFilter>();
     });

  4. Built-in attribute directly:
     [Authorize]
     [ResponseCache(Duration = 60)]

--------------------------------------------------------------------------------
3. MIDDLEWARE vs FILTERS — KEY DIFFERENCES
--------------------------------------------------------------------------------

  Factor                | Middleware                      | Filters
  ----------------------|---------------------------------|---------------------------
  Runs for              | Every HTTP request              | Only requests that reach
                        | (including static files,        | a matched controller action
                        | 404s, health checks)            |
  ----------------------|---------------------------------|---------------------------
  Position in pipeline  | Outer pipeline — runs before    | Inner pipeline — runs inside
                        | MVC framework                   | MVC framework
  ----------------------|---------------------------------|---------------------------
  Access to             | HttpContext only                 | HttpContext + action name,
                        | (no controller, no action)      | arguments, result, model
                        |                                 | binding state
  ----------------------|---------------------------------|---------------------------
  Where configured      | Program.cs (UseXxx methods)     | Attribute on action/controller
                        |                                 | or globally in AddControllers
  ----------------------|---------------------------------|---------------------------
  Scope                 | Application-wide always         | Can be scoped to one action,
                        |                                 | one controller, or global
  ----------------------|---------------------------------|---------------------------
  Order control         | Explicit order in Program.cs    | Fixed order by filter type
                        |                                 | (Auth → Resource → Action
                        |                                 | → Exception → Result)
  ----------------------|---------------------------------|---------------------------
  Good for              | Logging all requests, CORS,     | Validation, per-action auth,
                        | HTTPS redirect, rate limiting,  | action logging, exception
                        | compression, authentication     | handling per controller,
                        |                                 | response transformation
  ----------------------|---------------------------------|---------------------------
  Exception handling    | Catches ALL exceptions          | Only catches exceptions from
                        | (UseExceptionHandler)           | actions and action filters


--- When to Use Which ---

  Use Middleware when:
    - The concern applies to ALL requests regardless of whether they reach
      a controller (logging, HTTPS redirection, authentication, CORS, static
      files, rate limiting)
    - You need to run before routing even happens

  Use Filters when:
    - The concern is specific to controller actions (validate input, log
      action arguments, catch action-level exceptions, add response headers
      to API responses only)
    - You want to apply to a specific action or controller, not the entire app
    - You need access to MVC-specific information (action name, model state,
      action arguments)

--- Rule of Thumb ---

  If you need it for everything → Middleware
  If you need it only for controller actions → Filter
  If you need it for one controller or one action → Filter with an attribute

--------------------------------------------------------------------------------
SUMMARY TABLE
--------------------------------------------------------------------------------

  Term                    | What It Is
  ------------------------|------------------------------------------------------
  Middleware              | Component in the outer HTTP pipeline — runs for all
                          | requests, before MVC
  Terminal Middleware     | Middleware that short-circuits — returns without
                          | calling next()
  Custom Middleware       | Your own middleware class with InvokeAsync method
  Filter                  | Component in the inner MVC pipeline — runs only for
                          | matched controller actions
  Authorization Filter    | First to run — checks if the user is allowed access
  Resource Filter         | Wraps everything after auth — useful for early
                          | short-circuit (caching)
  Action Filter           | Runs before/after the action method — most common
                          | filter type
  Exception Filter        | Catches unhandled exceptions from actions only
  Result Filter           | Runs before/after the response is written
  Global Filter           | Filter registered in AddControllers — applies to all
                          | actions in the app
  UseExceptionHandler()   | Middleware for global exception handling (all errors)
  [Authorize]             | Built-in authorization filter attribute
