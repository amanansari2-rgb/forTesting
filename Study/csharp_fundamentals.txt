--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- Namespace ---

Technical definition:
  A namespace is a declarative region that provides scope to the identifiers
  (class names, method names, etc.) defined within it. It prevents naming
  conflicts when two classes in different libraries share the same name.

Simple way:
  A namespace is a folder for your code names. Two classes can both be called
  "Logger" as long as they live in different namespaces. You use the `using`
  keyword to bring a namespace into scope.

Example:
  using System.Collections.Generic;  // brings List<T>, Dictionary<T> into scope


--- Class vs Object ---

Simple way:
  A class is the blueprint. An object is the thing built from that blueprint.
  A class called Car describes what a car is. Each actual car you create
  (new Car()) is a separate object.


--- Method ---

Simple way:
  A method is a named set of instructions inside a class. You call it by name
  to run those instructions. It can take inputs and return a result.


--- Access Modifiers ---

Technical definition:
  Access modifiers are keywords that set the accessibility (visibility) of
  types and members. They control which parts of the program can read or
  call a given field, property, or method.

  public    → accessible from anywhere
  private   → accessible only within the same class
  protected → accessible within the class and any class that inherits from it
  internal  → accessible within the same assembly (.dll / .exe) only

Simple way:
  Access modifiers are the locks on your code. public means anyone can use it.
  private means only the class itself can use it. protected means the class and
  its children. internal means only code in the same project.


--- Interface ---

Technical definition:
  An interface is a contract — a type that defines a set of method and property
  signatures without any implementation. A class that implements an interface
  must provide the implementation for every member defined in it. A class can
  implement multiple interfaces.

Simple way:
  An interface says "any class that implements me MUST have these methods."
  It does not say how — just that they must exist. It is a guarantee, not code.

Example:
  interface IAnimal {
    void Speak();
  }

  class Dog : IAnimal {
    public void Speak() { Console.WriteLine("Woof"); }  // must implement Speak
  }


--- Delegate ---

Technical definition:
  A delegate is a type-safe function pointer. It holds a reference to a method
  that matches a specific signature (return type and parameters). It can be
  passed as an argument, stored in a variable, and invoked later.

Simple way:
  A delegate is a variable that holds a method. Instead of calling a method
  directly, you store it in a delegate and call it through the delegate. This
  lets you pass methods around like data.

Example:
  delegate int MathOperation(int a, int b);

  MathOperation add = (a, b) => a + b;
  int result = add(3, 5);  // result is 8

--------------------------------------------------------------------------------
1. OOP — OBJECT-ORIENTED PROGRAMMING
--------------------------------------------------------------------------------

Technical definition:
  Object-Oriented Programming is a programming paradigm that organises software
  around objects — self-contained units that combine data (fields/properties)
  and behaviour (methods). OOP is built on four core principles: Encapsulation,
  Inheritance, Polymorphism, and Abstraction. C# is a fully object-oriented
  language — almost everything in C# is an object.

Simple way:
  OOP is a way of structuring code around real-world things (objects). Instead
  of writing a long list of instructions, you model your program as a set of
  objects that each know their own data and what they can do.


--- Constructor ---

Technical definition:
  A constructor is a special method that is called automatically when an object
  is created with `new`. It has the same name as the class, no return type, and
  is used to initialise the object's fields to valid starting values.

Simple way:
  A constructor is the setup code that runs the moment you create an object.
  It makes sure the object starts in a valid, ready-to-use state.

Example:
  class Car {
    public string Brand;
    public int Year;

    public Car(string brand, int year) {  // constructor
      Brand = brand;
      Year  = year;
    }
  }

  Car myCar = new Car("Toyota", 2022);
  // Brand = "Toyota", Year = 2022 — set by the constructor


--- Pillar 1: Encapsulation ---

Technical definition:
  Encapsulation is the bundling of data (fields) and the methods that operate
  on that data into a single unit (a class), while restricting direct external
  access to the internal state. Access to internal data is provided only through
  controlled public methods or properties.

Simple way:
  Encapsulation hides the internal workings of a class. The outside world
  interacts with it only through what you intentionally expose. You protect
  the data from being accidentally changed in ways that would break the object.

Example:
  class BankAccount {
    private decimal balance;  // hidden — cannot be set from outside

    public void Deposit(decimal amount) {
      if (amount > 0) balance += amount;
    }

    public decimal GetBalance() {
      return balance;
    }
  }

  // balance is private — external code cannot write: account.balance = -9999


--- Properties (get / set) ---

Technical definition:
  A property is a class member that provides a controlled mechanism to read
  (get) or write (set) a private field. It looks like a public field from the
  outside but executes code (a getter or setter) when accessed. This is how
  encapsulation is typically implemented in C#.

Simple way:
  A property is a gatekeeper for a private field. The outside world thinks it
  is accessing a simple variable, but behind the scenes your code can validate
  or control the access.

Example:
  class Person {
    private int age;

    public int Age {
      get { return age; }
      set {
        if (value >= 0) age = value;  // reject negative age
      }
    }
  }


--- Pillar 2: Inheritance ---

Technical definition:
  Inheritance is the mechanism by which a class (derived/child class) acquires
  the members (fields, properties, methods) of another class (base/parent
  class) using the `:` syntax. The derived class can extend or override the
  behaviour of the base class. C# supports single inheritance for classes
  (one parent only) but multiple interface implementation.

Simple way:
  Inheritance lets a child class reuse everything the parent class has, without
  rewriting it. The child can also add its own members or change parent
  behaviour where needed.

Example:
  class Animal {
    public void Breathe() { Console.WriteLine("Breathing"); }
  }

  class Dog : Animal {   // Dog inherits Breathe() from Animal
    public void Bark() { Console.WriteLine("Woof"); }
  }

  Dog d = new Dog();
  d.Breathe();  // inherited from Animal
  d.Bark();     // defined in Dog


--- Pillar 3: Polymorphism ---

Technical definition:
  Polymorphism means "many forms." In C# it exists in two forms:
  1. Compile-time polymorphism (method overloading): multiple methods with
     the same name but different parameter lists in the same class.
  2. Runtime polymorphism (method overriding): a derived class provides its
     own implementation of a method defined in the base class using the
     `virtual` (base) and `override` (derived) keywords.

Simple way:
  Polymorphism lets the same method name work differently depending on context.
  Overloading means the same name with different inputs in the same class.
  Overriding means a child class replaces a parent class method's behaviour.


  Overloading example (compile-time — resolved when code is compiled):

    class Calculator {
      public int Add(int a, int b)          { return a + b; }
      public double Add(double a, double b) { return a + b; }
      // same name, different parameter types
    }

  Overriding example (runtime — resolved while program is running):

    class Animal {
      public virtual void Speak() { Console.WriteLine("..."); }
    }

    class Dog : Animal {
      public override void Speak() { Console.WriteLine("Woof"); }
    }

    class Cat : Animal {
      public override void Speak() { Console.WriteLine("Meow"); }
    }

    Animal a = new Dog();
    a.Speak();  // prints "Woof" — runtime decides which Speak() to call


--- Pillar 4: Abstraction ---

Technical definition:
  Abstraction is the principle of exposing only the essential behaviour of an
  object while hiding its implementation details. In C#, abstraction is
  achieved through abstract classes and interfaces.

Simple way:
  Abstraction means showing WHAT something does without showing HOW it does it.
  A method signature is the "what." The code inside the method is the "how."


  Abstract Class:

  Technical definition:
    An abstract class is a class that cannot be instantiated directly. It can
    contain abstract methods (declared with no body — just the signature) that
    derived classes MUST implement, as well as regular methods with full
    implementations that derived classes inherit.

  Simple way:
    An abstract class is a partial blueprint. It defines some required methods
    (without code) that child classes must fill in, and can also provide shared
    methods with code that children inherit.

  Example:
    abstract class Shape {
      public abstract double Area();        // no body — must be implemented
      public void Describe() {              // has body — inherited as-is
        Console.WriteLine("I am a shape");
      }
    }

    class Circle : Shape {
      private double radius;
      public Circle(double r) { radius = r; }
      public override double Area() { return Math.PI * radius * radius; }
    }


  Interface vs Abstract Class:

  Feature                    | Abstract Class          | Interface
  ---------------------------|-------------------------|-------------------------
  Can have method bodies?    | Yes                     | Yes (default methods,
                             |                         | C# 8+), but rarely used
  Can have fields?           | Yes                     | No
  Can have constructors?     | Yes                     | No
  Multiple inheritance?      | No (one parent only)    | Yes (many interfaces)
  Use when                   | Sharing code + contract | Pure contract only

2. EXTENSION METHODS
--------------------------------------------------------------------------------

Technical definition:
  An extension method is a static method defined in a static class that adds
  new functionality to an existing type without modifying the original type,
  deriving from it, or recompiling it. The first parameter of an extension
  method uses the `this` keyword followed by the type being extended. The
  method is then callable as if it were an instance method on that type.

Simple way:
  An extension method lets you add a new method to a type you do not own —
  like string, int, or a third-party class — without touching its source code.
  To the caller, it looks like a built-in method on that type, but it actually
  lives in a separate static class.

Syntax:
  public static class MyExtensions {
    public static ReturnType MethodName(this TargetType variable, ...) {
      // implementation
    }
  }

Example — adding a method to string:

  public static class StringExtensions {
    public static bool IsPalindrome(this string value) {
      string reversed = new string(value.Reverse().ToArray());
      return value == reversed;
    }
  }

  // Usage — looks like a string built-in method:
  bool result = "racecar".IsPalindrome();  // returns true
  bool result2 = "hello".IsPalindrome();   // returns false

  The string class was not changed. IsPalindrome is defined elsewhere but
  appears on string because of the `this string` parameter.

Rules:
  - The class containing the extension method MUST be static.
  - The extension method itself MUST be static.
  - The first parameter MUST use `this` before the target type.
  - You must import the namespace containing the extension class with `using`.

When to use:
  When you want to add utility methods to a type you cannot or should not
  modify — a .NET built-in type (string, int, List<T>), a third-party library
  type, or a sealed class. LINQ itself is built entirely from extension methods
  on IEnumerable<T>.

When NOT to use:
  If you own the class, add the method directly instead. Extension methods are
  for types you cannot touch.

3. GENERICS
--------------------------------------------------------------------------------

Technical definition:
  Generics allow classes, methods, interfaces, and delegates to be defined with
  a type placeholder (called a type parameter, conventionally named T) instead
  of a specific type. The actual type is supplied by the caller at compile time.
  This provides type safety without requiring casting, and avoids boxing/unboxing
  overhead for value types.

Simple way:
  Generics let you write one class or method that works with any data type,
  while still being type-safe. You write the logic once using T as a
  placeholder, and specify the real type (int, string, Car, etc.) when you
  use it.

Why generics exist — the problem without them:

  Without generics, you would use object type to hold anything:

    ArrayList list = new ArrayList();
    list.Add(1);
    list.Add("hello");   // no error — but mixing types is dangerous
    int x = (int)list[0]; // requires a cast — can fail at runtime

  With generics:

    List<int> list = new List<int>();
    list.Add(1);
    list.Add("hello");   // COMPILE ERROR — caught before the program runs
    int x = list[0];     // no cast needed — type is known


--- Generic Class ---

Technical definition:
  A generic class is a class defined with one or more type parameters. The
  type parameters act as placeholders that are replaced with real types when
  the class is instantiated.

Example:
  class Box<T> {
    private T value;

    public void Set(T val) { value = val; }
    public T Get()         { return value; }
  }

  Box<int>    intBox    = new Box<int>();
  Box<string> stringBox = new Box<string>();

  intBox.Set(42);
  stringBox.Set("hello");


--- Generic Method ---

Technical definition:
  A generic method is a method with its own type parameter, independent of
  whether the class itself is generic. The type parameter is declared after
  the method name inside angle brackets.

Example:
  public static T GetFirst<T>(T[] array) {
    return array[0];
  }

  int    first = GetFirst(new int[]    { 10, 20, 30 });  // T is int
  string word  = GetFirst(new string[] { "a", "b" });     // T is string


--- Generic Constraints (where T : ...) ---

Technical definition:
  Constraints restrict which types can be used as a type argument. They ensure
  the generic code can safely call certain methods or make certain assumptions
  about T.

  where T : class       → T must be a reference type
  where T : struct      → T must be a value type
  where T : new()       → T must have a public parameterless constructor
  where T : IInterface  → T must implement the specified interface
  where T : BaseClass   → T must be or inherit from BaseClass

Example:
  public static T CreateNew<T>() where T : new() {
    return new T();  // only works if T has a parameterless constructor
  }

4. COLLECTIONS
--------------------------------------------------------------------------------

Technical definition:
  Collections are classes in the BCL that store and manage groups of objects.
  The generic collections in System.Collections.Generic are the standard choice
  — they are type-safe, efficient, and cover most use cases.

Simple way:
  Collections are containers for multiple items. Each type of collection is
  designed for a specific pattern of use — ordered lists, key-value lookups,
  unique item sets, etc.


--- Array ---

Technical definition:
  An array is a fixed-size, zero-indexed sequence of elements of the same type,
  allocated as a contiguous block of memory. Its size is set at creation and
  cannot change.

Simple way:
  An array holds a fixed number of items in order. Fast to access by index,
  but you cannot add or remove items after creation.

Example:
  int[] numbers = new int[3] { 10, 20, 30 };
  Console.WriteLine(numbers[1]);  // 20


--- List<T> ---

Technical definition:
  List<T> is a generic, dynamically-resized array-backed list. It maintains
  insertion order, allows duplicates, and supports O(1) access by index.
  Internally it doubles its capacity when the current capacity is exceeded.

Simple way:
  List<T> is like an array that can grow or shrink. It is the most commonly
  used collection in C#. You can add, remove, and access items by index.

Example:
  List<string> names = new List<string>();
  names.Add("Aman");
  names.Add("Sara");
  names.Remove("Aman");
  Console.WriteLine(names[0]);  // "Sara"


--- Dictionary<TKey, TValue> ---

Technical definition:
  Dictionary<TKey, TValue> is a hash table-based collection of key-value pairs.
  Keys must be unique. Lookup, insertion, and deletion are O(1) on average
  because it uses the key's hash code to find the storage location directly.

Simple way:
  A dictionary stores data as pairs — a key and a value. You look up a value
  by its key, not by position. Like a real dictionary: look up a word (key)
  to get its definition (value). Keys must be unique.

Example:
  Dictionary<string, int> ages = new Dictionary<string, int>();
  ages["Aman"]  = 25;
  ages["Sara"]  = 30;
  Console.WriteLine(ages["Aman"]);  // 25


--- Queue<T> ---

Technical definition:
  Queue<T> is a FIFO (First-In, First-Out) collection. Items are added at
  the back (Enqueue) and removed from the front (Dequeue).

Simple way:
  A queue works like a line of people. The first person to join is the first
  to leave. You add to the back and remove from the front.

Example:
  Queue<string> queue = new Queue<string>();
  queue.Enqueue("Task 1");
  queue.Enqueue("Task 2");
  Console.WriteLine(queue.Dequeue());  // "Task 1"


--- Stack<T> ---

Technical definition:
  Stack<T> is a LIFO (Last-In, First-Out) collection. Items are added to the
  top (Push) and removed from the top (Pop).

Simple way:
  A stack works like a stack of plates. The last plate placed on top is the
  first one you take off. You add and remove from the same end (the top).

Example:
  Stack<string> stack = new Stack<string>();
  stack.Push("A");
  stack.Push("B");
  Console.WriteLine(stack.Pop());  // "B" — last in, first out


--- HashSet<T> ---

Technical definition:
  HashSet<T> is a collection that stores only unique elements. It uses a hash
  table internally. Adding a duplicate element is silently ignored. Lookup and
  insertion are O(1) on average.

Simple way:
  A HashSet is a bag that refuses duplicates. If you add the same item twice,
  it just keeps one. Useful when you need a list of unique items.

Example:
  HashSet<int> set = new HashSet<int>();
  set.Add(1);
  set.Add(2);
  set.Add(1);  // ignored — already in the set
  Console.WriteLine(set.Count);  // 2


--- IEnumerable<T> ---

Technical definition:
  IEnumerable<T> is the base interface for all generic collections. It exposes
  a single method: GetEnumerator(), which allows iteration with foreach. Any
  collection that implements IEnumerable<T> can be iterated and used with LINQ.

Simple way:
  IEnumerable<T> means "this thing can be looped through with foreach."
  List<T>, Array, Dictionary values, and LINQ query results all implement
  it. It is the common language that LINQ speaks to all collections.

5. LAMBDA EXPRESSIONS
--------------------------------------------------------------------------------

Technical definition:
  A lambda expression is an anonymous function — a function without a name —
  defined inline using the => (goes to) operator. It can be assigned to a
  delegate type (Func<>, Action<>, Predicate<>) or passed directly as an
  argument. Lambdas can capture variables from their surrounding scope
  (this is called a closure).

Simple way:
  A lambda is a small, nameless function you write inline where you need it,
  rather than defining a separate named method. Instead of declaring a whole
  method, you write the logic directly in one line.

Syntax:
  (parameters) => expression           // single expression, returns value
  (parameters) => { statements; }      // block body with multiple statements

Examples:
  Func<int, int> square = x => x * x;
  Console.WriteLine(square(5));  // 25

  Func<int, int, int> add = (a, b) => a + b;
  Console.WriteLine(add(3, 4));  // 7

  Action<string> print = message => Console.WriteLine(message);
  print("Hello");  // Hello


--- Func<T, TResult> ---

Technical definition:
  Func<T, TResult> is a built-in generic delegate type that represents a
  method that takes one or more input parameters and returns a value.
  The last type argument is always the return type.

Simple way:
  Func is a delegate for a method that takes something and returns something.

Example:
  Func<string, int> getLength = s => s.Length;
  Console.WriteLine(getLength("hello"));  // 5

  Func<int, int, bool> isGreater = (a, b) => a > b;


--- Action<T> ---

Technical definition:
  Action<T> is a built-in generic delegate type that represents a method that
  takes one or more parameters and returns void (nothing).

Simple way:
  Action is a delegate for a method that does something but returns nothing.

Example:
  Action<string> log = message => Console.WriteLine("[LOG] " + message);
  log("Application started");


--- Predicate<T> ---

Technical definition:
  Predicate<T> is a delegate that takes one input of type T and returns a bool.
  It represents a condition that is either true or false for a given value.

Simple way:
  A Predicate is a yes/no question about a value.

Example:
  Predicate<int> isEven = n => n % 2 == 0;
  Console.WriteLine(isEven(4));  // True
  Console.WriteLine(isEven(7));  // False

6. LINQ — LANGUAGE INTEGRATED QUERY
--------------------------------------------------------------------------------

Technical definition:
  LINQ (Language Integrated Query) is a set of extension methods and query
  syntax built into C# that allows querying any data source that implements
  IEnumerable<T> or IQueryable<T> using a uniform syntax. LINQ operators
  (Where, Select, OrderBy, etc.) are defined as extension methods in the
  System.Linq namespace and accept lambda expressions as arguments.

Simple way:
  LINQ lets you filter, sort, transform, and aggregate collections using
  clean, readable code. Instead of writing a loop and an if-statement to
  filter a list, you write one line with LINQ. It works the same way whether
  your data is a List, an array, a database table, or an XML file.


--- Two Syntaxes ---

  1. Method Syntax (fluent / lambda):
     Uses extension method calls chained together.
     This is the more common style.

  2. Query Syntax:
     Looks similar to SQL. Uses from, where, select keywords.
     Less common in modern C# but still valid.

Example — both do the same thing:

  List<int> numbers = new List<int> { 1, 2, 3, 4, 5, 6 };

  // Method syntax:
  var evens = numbers.Where(n => n % 2 == 0).ToList();

  // Query syntax:
  var evens = (from n in numbers where n % 2 == 0 select n).ToList();

  // Result: { 2, 4, 6 }


--- Deferred Execution ---

Technical definition:
  Most LINQ operators use deferred execution — the query is NOT evaluated when
  it is defined. It is evaluated only when the result is iterated (in a foreach
  loop) or when a terminating operator is called (ToList(), ToArray(), Count(),
  First(), etc.).

Simple way:
  Defining a LINQ query does not run it. It only runs when you actually need
  the results. This means if the source data changes between defining the query
  and reading the results, you get the updated data.

Example:
  var query = numbers.Where(n => n > 3);
  // query is not evaluated yet

  numbers.Add(10);
  // 10 is added BEFORE we iterate

  foreach (var n in query) { Console.Write(n + " "); }
  // Output: 4 5 6 10 — includes 10 because query ran now, not earlier


--- Common LINQ Operators ---

  Filtering:

    Where(x => condition)
      Returns elements that satisfy the condition.
      numbers.Where(n => n > 5)

  Transformation:

    Select(x => expression)
      Projects each element into a new form.
      names.Select(n => n.ToUpper())
      Returns a new sequence — does not change the original.

  Sorting:

    OrderBy(x => key)           → ascending
    OrderByDescending(x => key) → descending
    names.OrderBy(n => n.Length)

  Grouping:

    GroupBy(x => key)
      Groups elements by a key. Returns groups, each with a Key and its items.
      students.GroupBy(s => s.Grade)

  Element retrieval:

    First(x => condition)
      Returns the first matching element. Throws if none found.

    FirstOrDefault(x => condition)
      Returns the first matching element, or the default value (null / 0)
      if none found. Safer than First() when the result might be empty.

    Single(x => condition)
      Returns the one matching element. Throws if there is not exactly one.

    SingleOrDefault(x => condition)
      Returns the one matching element or default. Throws if more than one.

  Checking:

    Any(x => condition)
      Returns true if at least one element satisfies the condition.
      numbers.Any(n => n > 100)

    All(x => condition)
      Returns true if every element satisfies the condition.
      numbers.All(n => n > 0)

    Contains(value)
      Returns true if the sequence contains the specified value.

  Aggregation:

    Count()           → number of elements
    Sum(x => value)   → total of the values
    Min(x => value)   → smallest value
    Max(x => value)   → largest value
    Average(x => val) → arithmetic mean

  Distinct:

    Distinct()
      Removes duplicate elements.
      new[] { 1, 2, 2, 3 }.Distinct()  → { 1, 2, 3 }

  Paging:

    Take(n)           → returns first n elements
    Skip(n)           → skips first n elements, returns the rest
    Take and Skip are often combined for pagination.

  Conversion (terminates deferred execution):

    ToList()    → executes query, returns List<T>
    ToArray()   → executes query, returns T[]
    ToDictionary(key, value) → executes and returns Dictionary<TKey, TValue>


--- Chaining LINQ Operators ---

  LINQ operators can be chained because each one returns IEnumerable<T>.

Example:
  List<string> names = new List<string> { "Aman", "Sara", "Ali", "Zara", "Bob" };

  var result = names
    .Where(n => n.Length > 3)          // filter: names longer than 3 chars
    .OrderBy(n => n)                   // sort alphabetically
    .Select(n => n.ToUpper())          // transform to uppercase
    .ToList();                         // execute and store as List

  // Result: { "AMAN", "SARA", "ZARA" }


--- IEnumerable<T> vs IQueryable<T> ---

Technical definition:
  IEnumerable<T> executes the query in memory — all filtering and sorting
  happens in your application after data is loaded.
  IQueryable<T> translates the LINQ query into the native query language of
  the data source (e.g., SQL) and executes it there — only the filtered data
  is transferred to your application.

Simple way:
  With IEnumerable, your app loads all the data first, then filters it.
  With IQueryable (used with Entity Framework / databases), the filtering
  happens in the database — only the matching rows come back. IQueryable is
  more efficient for large data sources.

SUMMARY TABLE
--------------------------------------------------------------------------------

  Topic                    | What It Is
  -------------------------|----------------------------------------------------
  Encapsulation            | Hide internal data, expose through controlled members
  Inheritance              | Child class reuses and extends parent class members
  Polymorphism             | Same method name, different behaviour by type
  Abstraction              | Define what, not how — via abstract class / interface
  Abstract Class           | Partial blueprint — mix of abstract + real methods
  Interface                | Pure contract — defines what must exist, no code
  Extension Method         | Add methods to a type you cannot or should not modify
  Generics (T)             | Write once, use with any type — type-safe reuse
  List<T>                  | Dynamic ordered collection — most common collection
  Dictionary<TKey, TValue> | Key-value pairs — O(1) lookup by key
  Queue<T>                 | FIFO — first in, first out
  Stack<T>                 | LIFO — last in, first out
  HashSet<T>               | Unique items only — no duplicates allowed
  Lambda Expression        | Inline anonymous function using =>
  Func<T, TResult>         | Delegate for a method that takes input, returns value
  Action<T>                | Delegate for a method that takes input, returns void
  Predicate<T>             | Delegate for a yes/no condition on a value
  LINQ                     | Query collections with Where, Select, OrderBy, etc.
  Deferred Execution       | LINQ query runs only when results are actually read
  IQueryable<T>            | LINQ against databases — query runs in the database
