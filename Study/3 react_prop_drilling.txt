================================================================================
REACT — PROP DRILLING AND WHAT TO USE INSTEAD
================================================================================

--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- Component Tree ---

Technical definition:
  A component tree is the hierarchical structure of components in a React
  application. Each component can have a parent (the component that renders it)
  and children (the components it renders). The topmost component is called
  the root. Data in React naturally flows downward — from parent to child —
  through props. This direction is called "one-way data flow."

Simple way:
  Think of it like a family tree. There is a grandparent at the top, parents
  in the middle, and children at the bottom. In React, data can only be passed
  downward — from grandparent to parent to child — not upward.

Example of a tree:

  App                        ← root (top)
  └── Dashboard
      └── Sidebar
          └── UserProfile    ← deeply nested (bottom)


--- Consuming Data ---

Technical definition:
  A component "consumes" data when it actually reads and uses that data to
  render its output. An intermediate component that only receives a prop to
  pass it further down does NOT consume it.

Simple way:
  Consuming data means actually using it. If a component receives a prop but
  only passes it to its child without displaying or processing it, that
  component is not consuming it — it is just a middleman.


--- Intermediate Component ---

Technical definition:
  An intermediate component is any component that sits between the data source
  (where the data originates) and the component that actually needs the data.

Simple way:
  A middleman component. It receives data it does not need, only to pass it
  one level further down.

--------------------------------------------------------------------------------
1. WHAT IS PROP DRILLING?
--------------------------------------------------------------------------------

Technical definition:
  Prop drilling (also called "threading") is the process of passing props
  through one or more intermediate components that do not consume the data
  themselves, solely to deliver it to a deeply nested descendant component
  that does. The intermediate components are forced to accept and forward props
  they have no use for.

Simple way:
  Prop drilling is when you pass data from a top-level component all the way
  down through several middle components — even though those middle components
  do not need the data at all. They only hold it temporarily to hand it to the
  next one down the chain.

Example:

  Imagine the logged-in user's name needs to be shown inside UserProfile.
  The data lives in App. The path is:

  App → Dashboard → Sidebar → UserProfile

  Without any solution, you must pass the username prop at every level:

  function App() {
    const username = "Aman";
    return <Dashboard username={username} />;
  }

  function Dashboard({ username }) {
    // Dashboard does NOT use username, only passes it down
    return <Sidebar username={username} />;
  }

  function Sidebar({ username }) {
    // Sidebar does NOT use username, only passes it down
    return <UserProfile username={username} />;
  }

  function UserProfile({ username }) {
    // This is the ONLY component that actually needs username
    return <h1>Hello, {username}</h1>;
  }

  Dashboard and Sidebar are drilling the prop. They carry it but never use it.


--- Why Prop Drilling Is a Problem ---

  1. Unnecessary coupling:
     Intermediate components become aware of data they should not care about.
     If the data structure changes, you have to update every component in the
     chain even if they never used the data.

  2. Hard to maintain:
     If you rename a prop, you must find and update every component that
     passes it along — even the ones that never used it.

  3. Reduced reusability:
     A component that accepts a prop just to forward it cannot be used freely
     in a different part of the app without also providing that prop.

  4. Scalability:
     In a large application with many levels of nesting, this quickly becomes
     unmanageable.

--------------------------------------------------------------------------------
2. SOLUTION 1 — REACT CONTEXT API
--------------------------------------------------------------------------------

Technical definition:
  The React Context API is a built-in React mechanism for sharing data across
  the component tree without passing props at every level. It works through
  three parts:
    - React.createContext(): creates a Context object that holds the shared data
    - Context.Provider: a component that wraps part of the tree and supplies
      the data value to all descendants
    - useContext(Context): a Hook that any descendant component calls to read
      the current value from the nearest Provider above it in the tree

Simple way:
  Context is like a shared noticeboard. You pin data to the noticeboard at the
  top of the tree. Any component that wants the data can read it directly from
  the noticeboard — no matter how deep it is — without any middleman passing
  it down.


--- How Context Works (Step by Step) ---

  Step 1: Create the context
    This creates an empty "container" for your shared data.
	
import react, {createContext, useContext} from "react"

    const UserContext = createContext();

  Step 2: Wrap your tree with a Provider
    The Provider makes the data available to all components inside it.
    Only components inside this wrapper can access the data.

    function App() {
      const username = "Aman";
      return (
        <UserContext.Provider value={username}>
          <Dashboard />
        </UserContext.Provider>
      );
    }

  Step 3: Read the data in any descendant using useContext
    No props passed through Dashboard or Sidebar at all.

    function Dashboard() {
      return <Sidebar />;        // no prop needed
    }

    function Sidebar() {
      return <UserProfile />;    // no prop needed
    }

    function UserProfile() {
      const username = useContext(UserContext);  // reads directly
      return <h1>Hello, {username}</h1>;
    }

  Dashboard and Sidebar are no longer involved in passing the data.
  UserProfile reads it directly from the context.


--- When to Use Context ---

  Use Context when:
  - Data needs to be accessed by many components at different nesting levels
  - The data is relatively global to a section or the whole app
  - Common examples: logged-in user info, app theme (dark/light mode),
    selected language/locale, authentication status

  Do not overuse Context:
  - Context is not a replacement for all props. If only a parent and its
    direct child share data, passing a prop is simpler and clearer.
  - Every component that consumes a context re-renders when the context value
    changes. Putting frequently-changing data (like a counter that updates
    every second) in context can cause unnecessary re-renders across the tree.


--------------------------------------------------------------------------------
3. SOLUTION 2 — COMPONENT COMPOSITION
--------------------------------------------------------------------------------

Technical definition:
  Component Composition is the practice of passing components (as the children
  prop or as named props) rather than passing raw data. Instead of a parent
  sending data downward for a deeply nested component to use, the parent
  constructs the deeply nested component itself and passes it down as JSX.
  The intermediate components receive and render JSX they were given, without
  needing to know what data is inside it.

Simple way:
  Instead of handing data to a middleman and trusting it to pass it down, you
  build the final component at the top where the data lives, and hand the
  finished component to the middleman. The middleman just places it on screen
  without knowing what is inside.


--- Example ---

  Problem: Sidebar needs to render UserProfile, which needs username.
  Instead of drilling username through Sidebar, pass a ready-made component.

  function App() {
    const username = "Aman";
    return (
      <Sidebar content={<UserProfile username={username} />} />
    );
    // App constructs UserProfile here, where username is available
  }

  function Sidebar({ content }) {
    // Sidebar has no knowledge of username
    return <div className="sidebar">{content}</div>;
  }

  function UserProfile({ username }) {
    return <h1>Hello, {username}</h1>;
  }

  Sidebar receives a piece of JSX and places it. It never needs to know
  about username.


--- When to Use Component Composition ---

  Use Composition when:
  - The drilling involves only one or two levels
  - The intermediate component is a layout component (a wrapper, a panel,
    a sidebar) that does not care what it renders inside
  - You want to keep things simple without introducing Context

  Do not use it when:
  - Many different components across the tree all need the same data
  - The data needs to be accessed in deeply nested logic, not just rendered
    at one spot

--------------------------------------------------------------------------------
4. SOLUTION 3 — EXTERNAL STATE MANAGEMENT LIBRARIES
--------------------------------------------------------------------------------

Technical definition:
  External state management libraries (such as Redux, Zustand, or Jotai)
  maintain application state in a store that exists outside of the React
  component tree entirely. Any component can read from or write to this store
  directly, regardless of its position in the tree, without props or context.

Simple way:
  These libraries create a separate storage container that lives outside of
  all your components. Any component in the app can reach into that storage
  directly to get or update data. No prop passing, no context setup.

Common libraries:
  - Redux:  The most established option. Has strict rules about how data is
            read and updated. Best for large, complex applications with many
            developers.
  - Zustand: A simpler, lightweight alternative to Redux. Less boilerplate.
             Good for medium to large apps.
  - Jotai:  An atomic state library. Each piece of state is a separate "atom"
            any component can subscribe to individually.

When to use:
  - Large applications where many unrelated components need the same data
  - When you need advanced features like time-travel debugging (Redux DevTools)
  - When the Context API starts becoming difficult to manage due to too many
    separate contexts

When NOT to use:
  - Small or medium apps where Context or composition handles the problem
    cleanly. Adding Redux to a small app adds significant complexity for
    little gain. 

--------------------------------------------------------------------------------
5. SUMMARY — WHICH SOLUTION TO USE WHEN
--------------------------------------------------------------------------------

  Situation                                        | Best Solution
  -------------------------------------------------|------------------------
  Data shared between a parent and direct child    | Regular props (no issue)
  Data needed 2-3 levels deep in one place         | Component Composition
  Data needed by many components across the tree   | Context API
  Data changes rarely (theme, user info, locale)   | Context API
  Large app, complex shared state, many features   | Redux / Zustand
  Data changes very frequently (every keystroke)   | Local state + props
                                                   | (avoid Context for this)


--- One-Line Summary of the Problem and Each Solution ---

  Prop Drilling:         Passing data through middlemen who do not need it.
  Context API:           A shared noticeboard any component can read directly.
  Component Composition: Build the finished component where the data is,
                         pass the finished piece to the middleman.
  Redux / Zustand:       A storage container that lives outside the tree
                         entirely, accessible by any component.
