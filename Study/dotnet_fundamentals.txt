.NET FUNDAMENTALS — ARCHITECTURE, CLR, GARBAGE COLLECTION, CTS, SOFTWARE ENGINEERING
================================================================================

--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- What .NET Is ---

Technical definition:
  .NET is a free, open-source, cross-platform developer platform maintained by
  Microsoft. It provides a runtime environment, a base class library, and
  language compilers that allow code written in multiple languages (C#, F#,
  VB.NET) to be compiled, executed, and managed on the same underlying system.

Simple way:
  .NET is the environment your code runs in. You write code in a language like
  C#, and .NET provides all the tools to compile it, run it, manage its memory,
  and give it access to built-in functionality like file handling, networking,
  and collections.

.NET Implementations (versions you will hear about):

  .NET Framework   → The original Windows-only version (2002–2019). Legacy.
  .NET Core        → The cross-platform rewrite (2016). Runs on Windows, Linux,
                     macOS.
  .NET 5, 6, 7, 8+ → The unified platform that replaced both. Just called
                     ".NET" now. Cross-platform and actively maintained.


--- Source Code and Compilation ---

Technical definition:
  Compilation is the process of transforming human-readable source code (C#,
  for example) into a lower-level representation that a machine or runtime can
  execute. In .NET, this happens in two stages — first to an intermediate
  language, then to native machine code at runtime.

Simple way:
  You write code in English-like syntax. A compiler translates it into
  instructions a machine can understand. .NET does this in two steps rather
  than one, which gives it flexibility to run on different operating systems
  and processor types.


--- Runtime Environment ---

Technical definition:
  A runtime environment is software that provides the services needed to
  execute a compiled program. It handles memory, type checking, exception
  management, and communication with the operating system on behalf of the
  running program.

Simple way:
  A runtime is the system that runs your program after it has been compiled.
  It manages things your code should not have to worry about — like memory
  and errors — so you can focus on writing logic.


--- Managed vs Unmanaged Code ---

Technical definition:
  Managed code is code whose execution is supervised by the CLR. The runtime
  handles memory allocation, garbage collection, type safety, and exception
  handling. Unmanaged code (e.g., C, C++) runs directly on the OS without a
  runtime supervising it. The developer is responsible for allocating and
  freeing memory manually.

Simple way:
  Managed code has a supervisor (the CLR) watching over it, cleaning up memory,
  catching errors. Unmanaged code has no supervisor — if you forget to free
  memory, it leaks. .NET code is managed code.

--------------------------------------------------------------------------------
1. .NET ARCHITECTURE
--------------------------------------------------------------------------------

Technical definition:
  The .NET architecture is a layered system. At the top are .NET languages
  (C#, F#, VB.NET). Each language is compiled by its compiler into CIL
  (Common Intermediate Language), a CPU-independent instruction set. CIL is
  stored in an assembly (.dll or .exe file). At runtime, the CLR's JIT compiler
  converts CIL into native machine code for the specific CPU and OS it is
  running on. Underneath the CLR sits the BCL (Base Class Library), which
  provides the core types and APIs all .NET programs can use.

Simple way:
  Think of .NET as a stack of layers:

  [ Your Code — C#, F#, VB.NET ]
          ↓  compiled by language compiler
  [ CIL — Common Intermediate Language ]   ← stored in .dll / .exe
          ↓  converted at runtime by JIT
  [ Native Machine Code ]
          ↓  executed by
  [ Operating System / Hardware ]

  All .NET languages compile to the same middle layer (CIL). This is why
  C# code and F# code can work together in the same application — they both
  become the same thing (CIL) before running.


--- CIL (Common Intermediate Language) ---

Technical definition:
  CIL (also called MSIL — Microsoft Intermediate Language, or IL) is a
  CPU-independent, object-oriented instruction set that all .NET language
  compilers produce as output. It is not machine code — it cannot run directly
  on hardware. The CLR's JIT compiler converts it into native machine code at
  runtime.

Simple way:
  CIL is the middle step between your source code and machine code. It is like
  a universal translated version of your code that the CLR can then convert
  into the specific instructions the current machine needs.


--- Assembly (.dll / .exe) ---

Technical definition:
  An assembly is the compiled output of a .NET project. It is a file (either
  a .dll for libraries or a .exe for executable programs) that contains CIL
  code, metadata about the types and members defined in it, and a manifest
  describing the assembly's identity (name, version, dependencies).

Simple way:
  An assembly is the file your code becomes after compilation. It contains
  the translated (CIL) version of your code plus information about what is
  inside it. Other projects reference these files to use the code inside them.


--- BCL (Base Class Library) ---

Technical definition:
  The BCL is a set of fundamental classes and APIs built into .NET that provide
  core functionality — data types (string, int, bool), collections (List,
  Dictionary), file I/O, networking, threading, and more. All .NET languages
  and applications have access to it.

Simple way:
  The BCL is .NET's built-in toolbox. It contains ready-made code for common
  tasks so you do not have to write them yourself. Need to read a file? Sort
  a list? Make an HTTP request? The BCL has it.

--------------------------------------------------------------------------------
2. CLR — COMMON LANGUAGE RUNTIME
--------------------------------------------------------------------------------

Technical definition:
  The CLR (Common Language Runtime) is the virtual machine and execution
  engine of the .NET platform. It loads assemblies, JIT-compiles CIL into
  native machine code, manages memory via garbage collection, enforces type
  safety, handles exceptions, and provides services like thread management and
  security. It is what makes .NET code "managed."

Simple way:
  The CLR is the engine that runs your .NET program. It takes the compiled
  CIL code, converts it into real machine instructions, and then oversees the
  entire execution — cleaning up memory, catching errors, and managing threads
  — so you do not have to handle those things manually in your code.


--- JIT Compilation (Just-In-Time) ---

Technical definition:
  JIT compilation is the process by which the CLR converts CIL into native
  machine code at runtime, just before each method is executed for the first
  time. The compiled native code is cached in memory so the same method is
  not compiled twice in the same run of the program.

Simple way:
  Your code is stored as CIL (the middle-step language). When you actually
  call a method, the CLR converts that specific method into real machine code
  right then — "just in time" to run it. The next time you call it, the
  already-converted version is reused.

  This is different from languages like C, where everything is compiled into
  machine code before the program runs. JIT lets .NET run the same CIL on
  any platform by converting it to whatever machine code that platform needs.


--- What the CLR Manages ---

  1. Memory management  → Allocates memory for objects, runs GC to free it
  2. JIT compilation    → Converts CIL to native code at runtime
  3. Type safety        → Ensures you cannot treat a string as an integer, etc.
  4. Exception handling → Provides the try/catch/finally mechanism
  5. Thread management  → Manages multi-threading and synchronisation
  6. Security           → Enforces code access permissions

3. GARBAGE COLLECTION (GC)
--------------------------------------------------------------------------------

Technical definition:
  Garbage Collection is the CLR's automatic memory management system. It
  tracks all objects allocated on the managed heap. When memory is needed or
  a threshold is reached, the GC identifies objects that are no longer
  reachable by any live code path (unreachable objects), reclaims their memory,
  and compacts the remaining live objects to reduce heap fragmentation.
  Developers do not call free() or delete — the GC does it automatically.

Simple way:
  When your program creates objects, they take up memory. When you are done
  with them, that memory needs to be freed. In C or C++, you do this manually.
  In .NET, the GC watches which objects are still being used. When an object
  is no longer referenced by anything in your code, the GC reclaims that memory
  automatically. You never have to delete objects yourself.


--- Large Object Heap (LOH) ---

Technical definition:
  The Large Object Heap is a separate region of the managed heap for objects
  that are 85,000 bytes or larger (e.g., large arrays). LOH objects are
  collected with Gen 2 and are not compacted by default, because moving large
  objects in memory is expensive.

Simple way:
  Very large objects get their own separate area of memory (the LOH) because
  moving them around during compaction would be too slow. They are treated
  differently from regular objects.


--- IDisposable and the Dispose Pattern ---

Technical definition:
  IDisposable is an interface with a single method: Dispose(). It is used for
  objects that hold unmanaged resources (file handles, database connections,
  network sockets) that the GC cannot automatically release. Calling Dispose()
  explicitly releases those resources immediately rather than waiting for GC.
  The using statement in C# calls Dispose() automatically when the block ends.

Simple way:
  The GC handles managed memory automatically, but it cannot close a file or
  a database connection — those are unmanaged resources. IDisposable lets a
  class define cleanup code for those resources. The using statement ensures
  Dispose() is always called, even if an exception occurs.

Example:
  using (var file = new StreamReader("data.txt"))
  {
    string content = file.ReadToEnd();
  } // Dispose() is called automatically here — file handle is released

4. CTS — COMMON TYPE SYSTEM
--------------------------------------------------------------------------------

Technical definition:
  The CTS (Common Type System) is the specification that defines how types are
  declared, used, and managed in the CLR. It establishes the rules that all
  .NET languages must follow so that types written in one language (e.g., C#)
  can be used by another (e.g., VB.NET or F#) without any conversion. It
  defines two categories of types: value types and reference types.

Simple way:
  The CTS is the rulebook for types in .NET. Because C#, F#, and VB.NET all
  follow the same CTS rules, a class written in C# can be used directly in
  an F# project. They all speak the same type language.


--- Boxing and Unboxing ---

Technical definition:
  Boxing is the process of converting a value type into a reference type
  (object) by wrapping it in a heap-allocated object. Unboxing is the reverse:
  extracting the value type back out of the object. Both operations have a
  performance cost — boxing involves a heap allocation.

Simple way:
  Sometimes .NET needs to treat a value type (like an int) as an object.
  To do this, it "boxes" it — wraps it in a small object on the heap.
  Unboxing is unwrapping it back. This conversion has a cost, so you want
  to avoid doing it in performance-sensitive code in large loops.

Example:
  int x = 42;
  object boxed = x;      // boxing   — int wrapped into an object on the heap
  int unboxed = (int)boxed;  // unboxing — extracted back out


--- CLS (Common Language Specification) ---

Technical definition:
  The CLS is a subset of the CTS. It defines the minimum set of features a
  .NET language must support to guarantee interoperability with other
  .NET languages. If a library is CLS-compliant, it can be used by any
  .NET language, not just the one it was written in.

Simple way:
  The CTS is the full rulebook. The CLS is the agreed minimum — the features
  every .NET language must support so they can all work together. If you write
  a library and mark it as CLS-compliant, you are guaranteeing it works from
  any .NET language.

5. SOFTWARE ENGINEERING
--------------------------------------------------------------------------------

Technical definition:
  Software Engineering is the discipline of applying systematic, measurable,
  and repeatable engineering principles to the design, development, testing,
  maintenance, and documentation of software. It covers the full software
  development lifecycle (SDLC) and emphasises correctness, reliability,
  maintainability, and scalability of software systems.

Simple way:
  Software engineering is the practice of building software in a structured,
  reliable, and maintainable way — not just making it work, but making it
  well-organised, testable, and easy to change over time.


--- How .NET Supports Software Engineering ---

  Strong typing (enforced by CTS):
    Technical: The CLR enforces type safety at compile time and runtime,
    preventing type mismatches that cause undefined behaviour.
    Simple: The compiler catches type errors before your program even runs.
    You cannot accidentally treat a number as a string. Fewer bugs reach
    production.

  Managed Memory (GC):
    Technical: Automatic memory management eliminates entire categories of
    bugs common in unmanaged languages — memory leaks, dangling pointers,
    double-free errors.
    Simple: You do not have to manually track and release memory. This removes
    a large class of hard-to-find bugs from .NET applications.

  Object-Oriented Programming (OOP):
    Technical: .NET fully supports encapsulation, inheritance, and polymorphism
    through the CTS class system. These allow code to be modelled as interacting
    objects with clear responsibilities.
    Simple: OOP lets you structure code into organised, reusable pieces with
    clear boundaries, making large programs easier to build and maintain.

  SOLID Principles (applied in .NET):
    These are five design principles widely applied in .NET development:

    S — Single Responsibility Principle:
      A class should have one job and one reason to change.

    O — Open/Closed Principle:
      A class should be open for extension but closed for modification.
      Add new behaviour through inheritance or interfaces, not by editing
      existing code.

    L — Liskov Substitution Principle:
      A subclass should be usable wherever its parent class is expected,
      without breaking the program.

    I — Interface Segregation Principle:
      Do not force a class to implement methods it does not need. Split large
      interfaces into smaller, specific ones.

    D — Dependency Inversion Principle:
      High-level modules should not depend on low-level modules directly.
      Both should depend on abstractions (interfaces). This is the foundation
      of Dependency Injection, which is built into .NET.

  Base Class Library (BCL):
    Technical: The BCL provides tested, standardised implementations of common
    functionality so developers do not rewrite them from scratch.
    Simple: .NET ships with a large library of ready-made tools — collections,
    file handling, cryptography, HTTP clients, etc. You reuse proven code
    instead of building it yourself, reducing bugs and saving time.

  Testing Support:
    Technical: .NET supports unit testing through frameworks like xUnit, NUnit,
    and MSTest. The platform's design (interfaces, dependency injection) makes
    code easier to test in isolation.
    Simple: .NET is designed in a way that makes it straightforward to write
    automated tests for your code, which is a core software engineering practice.

  Cross-Language Interoperability:
    Technical: Because all .NET languages compile to CIL and share the CTS,
    a library written in C# can be used directly from VB.NET or F# without
    any translation layer.

    Simple: Teams using different .NET languages can share code freely. A C#
    library works in an F# project without any changes.
