REACT COMPONENTS — TYPES, USAGE, AND CONTROLLED vs UNCONTROLLED

--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- props (Reminder) ---

Technical definition:
  Read-only data passed from a parent component to a child component via
  attributes. A component cannot modify its own props.
Understand state in the youtube


Simple way:
  Props are inputs. A parent sends data to a child component through props,
  and the child uses that data to decide what to display. The child cannot
  change the data it receives.

--- state (Reminder) ---

Technical definition:
  Mutable data owned and managed by a component itself. When state changes,
  React schedules a re-render of that component.

Simple way:
  State is a component's own memory. The component can change it, and when it
  does, the screen updates automatically.

--- ref ---

Technical definition:
  A ref (reference) is an object created with useRef() that holds a mutable
  value in its .current property. It persists across re-renders but changing
  it does NOT trigger a re-render. It is commonly used to directly access a
  DOM element.

Simple way:
  A ref is like a direct pointer to an HTML element or a value that you want
  to read/write without causing the component to redraw. Think of it as a
  sticky note attached to an element — you can read or update it anytime.

1. WHAT IS A COMPONENT? (Quick Recap)
--------------------------------------------------------------------------------

Technical definition:
  A component is an independent, reusable unit of UI in React. It is a
  JavaScript function (or class) that accepts props as input and returns JSX
  describing what should appear on screen.

Simple way:
  A component is a building block of your page. Each component is responsible
  for one piece of the UI — a button, a form, a navigation bar. You build many
  small components and combine them to make a full page.

--------------------------------------------------------------------------------
2. TYPES OF COMPONENTS
--------------------------------------------------------------------------------

There are two types based on HOW they are written:

  1. Function Component
  2. Class Component

--- 2.1 Function Component ---

Technical definition:
  A Function Component is a plain JavaScript function that accepts a props
  object as its argument and returns JSX. Since React 16.8, function components
  can use Hooks (useState, useEffect, etc.) to manage state and side effects,
  making them fully capable of doing everything a class component can do.
  This is the current standard in React.

Simple way:
  A function component is just a JavaScript function. It takes some input
  (props), and returns what to show on screen. Most React code written today
  uses this type.

Where it is used:
  Everywhere in modern React. New projects, new features, new components —
  all written as function components.

Example:
  function Greeting(props) {
    return <h1>Hello, {props.name}</h1>;
  }

  Usage: <Greeting name="Aman" />
  Output on screen: Hello, Aman


--- 2.2 Class Component ---

Technical definition:
  A Class Component is an ES6 class that extends React.Component. It must
  implement a render() method that returns JSX. State is managed via
  this.state and updated with this.setState(). Lifecycle methods like
  componentDidMount() and componentDidUpdate() are used to handle side effects.

Simple way:
  The older way of writing components. Before 2019, this was the only way to
  have state inside a component. It requires more boilerplate (extra code)
  compared to function components. You will mostly encounter this in older
  codebases, not in new projects.

Where it is used:
  Legacy React codebases (code written before React 16.8). New code rarely
  uses class components anymore.

Example:
  class Greeting extends React.Component {
    render() {
      return <h1>Hello, {this.props.name}</h1>;
    }
  }

Key difference in one line:
  Function Component uses Hooks for state.
  Class Component uses this.state and this.setState() for state.

3. COMPONENT ROLES — WHAT JOB DOES A COMPONENT DO?
--------------------------------------------------------------------------------

Beyond how they are written, components also differ by their ROLE in the app.
These are not enforced by React — they are patterns developers follow to keep
code organized.

--- 3.1 Presentational Component (also called Dumb or Stateless Component) ---

Technical definition:
  A Presentational Component is concerned only with how things look. It
  receives data through props and renders UI. It does not fetch data, does not
  contain business logic, and typically has no state (or only UI-related state
  like a toggle). It is highly reusable because it has no knowledge of where
  the data comes from.

Simple way:
  A presentational component is purely for display. It takes data as props and
  shows it. It does not care where the data came from or what to do with it
  logically. Think of it as a template.

Where it is used:
  Reusable UI elements — buttons, cards, badges, avatars, list items, modals.
  Anything that can be reused across many pages.

Example:
  A <UserCard> component that takes a name and email as props and displays them.
  It does not fetch the user — it just shows what it is given.

  function UserCard(props) {
    return (
      <div>
        <h2>{props.name}</h2>
        <p>{props.email}</p>
      </div>
    );
  }


--- 3.2 Container Component (also called Smart or Stateful Component) ---

Technical definition:
  A Container Component handles data fetching, business logic, and state
  management. It is responsible for WHAT data is shown, not how it looks.
  It usually passes data down to presentational components as props.

Simple way:
  A container component does the work behind the scenes — it fetches data from
  an API, manages state, and then hands the results to a presentational
  component to display. It does not care much about how things look.

Where it is used:
  Pages, feature sections, or any component that needs to talk to an API or
  manage complex state. For example, a UserList page that fetches a list of
  users and passes them to individual <UserCard> components.

Example:
  function UserListPage() {
    const [users, setUsers] = useState([]);

    useEffect(() => {
      fetch("/api/users")
        .then(res => res.json())
        .then(data => setUsers(data));
    }, []);

    return (
      <div>
        {users.map(user => (
          <UserCard key={user.id} name={user.name} email={user.email} />
        ))}
      </div>
    );
  }

  Here, UserListPage is the container. UserCard is the presentational component.

Note:
  This separation is a pattern, not a strict rule. In smaller apps, components
  often do both jobs at once and that is completely fine.

4. CONTROLLED vs UNCONTROLLED COMPONENTS
--------------------------------------------------------------------------------

Note on your question:
  "Structured and unstructured component" is not a standard React term. The
  actual React concept that most closely matches this idea is
  CONTROLLED and UNCONTROLLED components. These describe how form inputs
  (text fields, checkboxes, dropdowns, etc.) manage their data.

--- 4.1 Controlled Component ---

Technical definition:
  A Controlled Component is a form element (input, textarea, select) whose
  value is driven entirely by React state. The component's state is the single
  source of truth for the input's value. Every change to the input calls a
  state-updating function (onChange handler), keeping the DOM and the React
  state in sync at all times.

Simple way:
  React is in full control of the input. When you type into a text box, React
  state updates immediately, and the text box displays whatever React says.
  Nothing happens in the input that React does not know about.

Where it is used:
  - Forms where you need to validate input on every keystroke
  - Forms where you need to submit or read the data at any point
  - When multiple fields depend on each other's values
  Most form handling in React uses controlled components.

Example:
  function LoginForm() {
    const [email, setEmail] = useState("");

    return (
      <input
        type="text"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
      />
    );
  }

  The input's value is always equal to the email state variable.
  React controls it.


--- 4.2 Uncontrolled Component ---

Technical definition:
  An Uncontrolled Component is a form element that manages its own value
  internally in the DOM, as standard HTML does by default. React does not
  track its value through state. Instead, a ref is used to directly read the
  DOM element's current value when needed (e.g., on form submit).

Simple way:
  The input manages itself, like a regular HTML input would. React does not
  watch it. You only check what was typed when you actually need it (like when
  the form is submitted), by reading the ref.

Where it is used:
  - Simple forms where you only need the value on submit, not on every keystroke
  - Integrating with non-React code or libraries that manage their own DOM
  - File inputs (<input type="file">) — these are always uncontrolled because
    their value cannot be set programmatically for security reasons

Example:
  function LoginForm() {
    const emailRef = useRef(null);

    function handleSubmit() {
      console.log(emailRef.current.value); // read only when submitted
    }

    return (
      <div>
        <input type="text" ref={emailRef} />
        <button onClick={handleSubmit}>Submit</button>
      </div>
    );
  }

  React does not know what the user is typing until handleSubmit is called.


--- Side-by-side comparison ---

  Feature                   | Controlled          | Uncontrolled
  --------------------------|---------------------|-------------------
  Data lives in             | React state         | The DOM itself
  How you read the value    | From state variable | Via ref
  Tracks every keystroke?   | Yes                 | No
  Validation on each key?   | Easy                | Harder
  Less React code needed?   | No                  | Yes (simpler setup)
  Standard for most forms?  | Yes                 | Only in specific cases

--------------------------------------------------------------------------------
5. SUMMARY — WHICH TYPE GOES WHERE
--------------------------------------------------------------------------------

  Component Type            | When to Use
  --------------------------|----------------------------------------------------
  Function Component        | Default choice for everything in modern React
  Class Component           | Only when maintaining old (legacy) code
  Presentational Component  | Reusable display elements (buttons, cards, items)
  Container Component       | Pages or features that fetch data or hold logic
  Controlled Component      | Forms where React needs to track every change
  Uncontrolled Component    | Simple forms, file inputs, or non-React integration
