--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- What a Hook Is ---

Technical definition:
  A Hook is a special function provided by React that lets function components
  "hook into" React features — such as state, lifecycle behaviour, and context
  — that were previously only available in class components. All built-in React
  Hooks start with the prefix "use".

Simple way:
  Hooks are tools React gives you to add features to your function components.
  Before Hooks existed, if you needed state or lifecycle behaviour, you had to
  write a class component. Hooks removed that requirement.


--- Rules of Hooks ---

Technical definition:
  React enforces two rules for Hooks:
  1. Only call Hooks at the top level of a function component or a custom Hook.
     Do not call Hooks inside loops, conditions, or nested functions.
  2. Only call Hooks inside React function components or custom Hooks.
     Do not call them in regular JavaScript functions.
  These rules exist because React tracks Hooks by the order they are called.
  If that order changes between renders, React's internal state gets out of sync.

Simple way:
  Always call Hooks at the top of your component, every time, in the same order.
  Never put a Hook inside an if-statement or a loop.

Example of what NOT to do:
  if (isLoggedIn) {
    const [name, setName] = useState(""); // WRONG — Hook inside a condition
  }


--- Re-render ---

Technical definition:
  A re-render is when React calls your component function again to produce a new
  Virtual DOM snapshot. This happens when the component's state or props change.
  React then diffs the new snapshot against the previous one and updates only
  the changed parts of the real DOM.

Simple way:
  A re-render is React re-running your component function to refresh what is
  shown on screen. It happens automatically when something the component depends
  on changes.


--- Side Effect ---

Technical definition:
  A side effect is any operation that reaches outside the component's render
  function to interact with something external — such as fetching data from an
  API, setting a timer, subscribing to an event, or manually modifying the DOM.
  These operations are called "side" effects because they are not part of the
  pure input-to-output rendering process.

Simple way:
  A side effect is anything your component does beyond just returning JSX.
  Fetching data, starting a timer, or listening for a window resize event are
  all side effects — they interact with the outside world.


--- Memoization ---

Technical definition:
  Memoization is an optimization technique where the result of a function call
  (or the function reference itself) is cached. On subsequent calls, if the
  inputs have not changed, the cached result is returned instead of recomputing.

Simple way:
  Memoization means remembering a result so you do not have to recalculate it
  if the input is the same. React uses this to skip unnecessary work during
  re-renders.


--- Referential Equality ---

Technical definition:
  In JavaScript, objects and functions are compared by reference, not by value.
  Two objects with identical contents are NOT equal unless they point to the
  same location in memory. This matters in React because every time a component
  re-renders, new functions and objects are created — even if their content
  did not change — which can cause child components to re-render unnecessarily.

Simple way:
  In JavaScript, { a: 1 } === { a: 1 } is FALSE, even though the content looks
  the same, because they are two separate objects in memory. In React, this
  means a function defined inside a component is "new" on every render. This
  can cause problems when you pass it as a prop to a child.

Example:
  const a = { name: "Aman" };
  const b = { name: "Aman" };
  console.log(a === b); // false — different objects in memory

  const c = a;
  console.log(a === c); // true — same reference


--- Dependency Array ---

Technical definition:
  A dependency array is the second argument passed to useEffect, useMemo, and
  useCallback. It is an array of values that React watches. The Hook's callback
  only re-runs when at least one value in this array has changed since the last
  render (compared using referential equality).

Simple way:
  The dependency array tells React: "only run this again if one of these
  values has changed." If the array is empty, it runs only once. If it is
  omitted, it runs on every render.

  []         → run once (on mount only)
  [a, b]     → run whenever a or b changes
  (omitted)  → run on every render

1. useState
--------------------------------------------------------------------------------

Technical definition:
  useState is a Hook that adds a state variable to a function component. It
  returns a pair: the current state value, and a setter function to update it.
  Calling the setter function schedules a re-render with the new state value.
  The initial value is only used on the first render.

Simple way:
  useState gives your component memory. You create a variable and a way to
  change it. When you change it using the setter, React updates the screen.

Syntax:
  const [value, setValue] = useState(initialValue);

Returns:
  An array of exactly two items — React calls this a "tuple":
  [0] state     → the current value of the state variable.
                  Type is whatever type you passed as initialValue.
  [1] setState  → a setter function. Call it with a new value to update
                  state and trigger a re-render.
                  Accepts either a direct value: setState(newValue)
                  or a function:               setState(prev => prev + 1)

  const [count, setCount] = useState(0);
  //     ↑ state value     ↑ setter function

Example:
  function Counter() {
    const [count, setCount] = useState(0);

    return (
      <div>
        <p>Count: {count}</p>
        <button onClick={() => setCount(count + 1)}>Add</button>
      </div>
    );
  }

  Every time the button is clicked, setCount updates count and React re-renders
  the component showing the new number.

Important note — functional update:
  If the new state depends on the previous state, pass a function to the setter
  instead of a direct value. This ensures you always work with the latest state.

  setCount(prev => prev + 1);  // safe
  setCount(count + 1);         // can be stale in some edge cases

When to use:
  Any time a component needs to track a value that can change — a form input,
  a toggle, a counter, a list of items.

2. useEffect
--------------------------------------------------------------------------------

Technical definition:
  useEffect is a Hook that lets you perform side effects in function components.
  It runs after React has committed the render to the DOM. It accepts a callback
  function and an optional dependency array. The callback can optionally return
  a cleanup function, which React calls before the next effect runs or when the
  component is removed from the DOM (unmounts).

Simple way:
  useEffect is where you put work that happens after the page updates — like
  fetching data, setting up a timer, or subscribing to something. React runs
  it after rendering, not during. The cleanup function is how you undo that
  work when it is no longer needed.

Syntax:
  useEffect(() => {
    // side effect here

    return () => {
      // cleanup here (optional)
    };
  }, [dependencies]);

Returns:
  useEffect itself returns nothing (void).
  The CALLBACK you pass inside it can optionally return a cleanup function.
  That cleanup function also returns nothing (void).

  Do NOT try to use the return value of useEffect — it has none.
  The only "return" involved is the cleanup function inside the callback:

  useEffect(() => {
    // ... setup work
    return () => { /* cleanup */ };  // ← this is the cleanup, not a return value
  }, []);

Three behaviours based on the dependency array:

  1. No dependency array — runs after every single render
     useEffect(() => { console.log("ran"); });

  2. Empty array — runs once after the first render only (on mount)
     useEffect(() => { fetchData(); }, []);

  3. With dependencies — runs after first render, then again when deps change
     useEffect(() => { fetchUser(userId); }, [userId]);

Example — fetching data:
  function UserProfile({ userId }) {
    const [user, setUser] = useState(null);

    useEffect(() => {
      fetch(`/api/users/${userId}`)
        .then(res => res.json())
        .then(data => setUser(data));
    }, [userId]);
    // re-fetches whenever userId changes

    return <div>{user ? user.name : "Loading..."}</div>;
  }

Example — cleanup (timer):
  useEffect(() => {
    const timer = setInterval(() => {
      console.log("tick");
    }, 1000);

    return () => clearInterval(timer); // cleanup when component unmounts
  }, []);

When to use:
  Fetching data, setting up event listeners, integrating with third-party
  libraries, starting or clearing timers.

When NOT to use:
  Do not use useEffect to transform data that can be computed directly during
  render. If you can calculate a value from props or state without a side
  effect, just compute it inline — no useEffect needed.

3. useContext
--------------------------------------------------------------------------------

Technical definition:
  useContext is a Hook that reads and subscribes to a React Context. It accepts
  a Context object (created by React.createContext) and returns the current
  context value provided by the nearest matching Provider above the component
  in the tree. When the Provider's value changes, the component re-renders with
  the new value.

Simple way:
  useContext is how a component reads data from a Context (the shared
  noticeboard described in the prop drilling document). Instead of receiving
  data through props, the component reaches up to the nearest Provider and
  reads the value directly.

Syntax:
  const value = useContext(MyContext);

Returns:
  The current context value — whatever was passed to the nearest
  Provider's value prop above this component in the tree.
  The type is exactly whatever type the context was created with.

  If no Provider is found above, it returns the default value that
  was set when the context was created with React.createContext(defaultValue).

  const theme = useContext(ThemeContext);
  //    ↑ whatever ThemeContext.Provider passed as value="dark"

Example:
  const ThemeContext = React.createContext();

  function App() {
    return (
      <ThemeContext.Provider value="dark">
        <Page />
      </ThemeContext.Provider>
    );
  }

  function Page() {
    return <Toolbar />;  // does not need to know about theme
  }

  function Toolbar() {
    const theme = useContext(ThemeContext);
    return <p>Current theme: {theme}</p>;
  }

When to use:
  Reading shared data like the logged-in user, app theme, or selected language
  — data that many components across the tree need.

When NOT to use:
  If only a parent and its direct child share the data, a regular prop is
  cleaner. Also avoid putting frequently-changing values in context as it will
  cause every consumer to re-render on each change.

4. useRef
--------------------------------------------------------------------------------

Technical definition:
  useRef returns a mutable object with a single property: .current. This object
  persists for the full lifetime of the component. Mutating .current does NOT
  trigger a re-render. useRef is primarily used for two things:
  1. Storing a reference to a DOM element
  2. Storing a mutable value that needs to persist across renders without
     causing re-renders when it changes

Simple way:
  useRef is a box that holds a value. Unlike state, changing what is in the
  box does NOT refresh the screen. It is used either to point directly at an
  HTML element (so you can focus it or read its value), or to store a number
  or value you want to remember without triggering a re-render.

Syntax:
  const ref = useRef(initialValue);
  ref.current   // how you read or write the value

Returns:
  A plain JavaScript object with one property: { current: T }
  The .current property starts as the initialValue you passed in.
  React keeps this same object alive for the entire life of the component.

  Two common shapes depending on usage:

  When used to store a value:
    const countRef = useRef(0);
    countRef.current  // → 0 (or whatever you last set it to)

  When attached to a DOM element via the ref attribute:
    const inputRef = useRef(null);
    // after render: inputRef.current → the actual <input> DOM node
    // before render or if element not mounted: inputRef.current → null

  The object itself never changes — React always returns the same reference.
  Only .current changes when you assign to it.

Example 1 — accessing a DOM element (focus an input):
  function SearchBar() {
    const inputRef = useRef(null);

    function handleClick() {
      inputRef.current.focus(); // directly interacts with the DOM element
    }

    return (
      <div>
        <input ref={inputRef} type="text" />
        <button onClick={handleClick}>Focus Input</button>
      </div>
    );
  }

Example 2 — storing a value without re-rendering (tracking a timer ID):
  function Timer() {
    const timerRef = useRef(null);

    function start() {
      timerRef.current = setInterval(() => console.log("tick"), 1000);
    }

    function stop() {
      clearInterval(timerRef.current);
    }

    return (
      <div>
        <button onClick={start}>Start</button>
        <button onClick={stop}>Stop</button>
      </div>
    );
  }

  If you stored the timer ID in state, updating it would cause a re-render.
  Using a ref avoids that unnecessary re-render.

When to use:
  Focusing or measuring DOM elements, storing timer IDs, storing the previous
  value of a state variable, or any value that changes but should NOT cause
  a re-render.

--- Q: Why not just use a plain variable (let or const) instead of useRef? ---

This is a valid question. If useRef is used to store values without triggering
a re-render, why not just write let timerId = null inside the component?

The answer comes down to one core fact about React:

  Every time a component re-renders, React calls the entire component
  function again from scratch. Every variable declared inside the function
  is reset to its initial value on that re-render.

So a plain variable does NOT survive a re-render. useRef does.

  Plain variable — reset on every re-render:

    function Timer() {
      let timerId = null;  // wiped back to null on every re-render

      function start() {
        timerId = setInterval(() => console.log("tick"), 1000);
        // timerId holds the ID here... but gone on next re-render
      }

      function stop() {
        clearInterval(timerId);  // timerId is null — does nothing
      }
    }

    The moment any state change causes a re-render, timerId resets to null.
    The timer ID saved in start() is lost.

  useRef — survives every re-render:

    function Timer() {
      const timerRef = useRef(null);
      // React creates this object once and hands back the SAME object
      // on every re-render. It is never reset.

      function start() {
        timerRef.current = setInterval(() => console.log("tick"), 1000);
      }

      function stop() {
        clearInterval(timerRef.current);  // still has the ID — works correctly
      }
    }

What about declaring the variable OUTSIDE the component?

    let timerId = null;  // outside the component function

    function Timer() { ... }

  This does persist across re-renders, but it creates a different problem:
  the variable is now shared across ALL instances of the component. If two
  <Timer /> components are on the page at the same time, both read and write
  to the same timerId. One instance can overwrite the other's timer ID.

  useRef is scoped to one specific component instance. Each <Timer /> gets
  its own ref object, completely separate from any other instance.

Summary of the three options:

  Option                         | Survives re-render? | Per instance?
  -------------------------------|---------------------|---------------
  let/const inside component     | No — reset each time | Yes
  Variable outside component     | Yes                  | No — shared
  useRef                         | Yes                  | Yes          ✓

useRef is the only option that is both stable across re-renders AND
isolated to the specific component instance it belongs to.

5. useMemo
--------------------------------------------------------------------------------

Technical definition:
  useMemo is a Hook that memoizes the return value of a function. It accepts
  a function and a dependency array. It runs the function on the first render
  and caches its return value. On subsequent renders, if the dependencies have
  not changed, it returns the cached value instead of re-running the function.
  If dependencies change, it re-runs and caches the new value.

Simple way:
  useMemo remembers the RESULT of a calculation. If the inputs to the
  calculation have not changed since the last render, React skips recalculating
  and returns the previously saved result. Use it for calculations that are
  expensive to run every render.

Syntax:
  const result = useMemo(() => computeSomething(a, b), [a, b]);

Returns:
  The memoized result — whatever your factory function returns.
  The type is exactly the return type of the function you pass in.

  On first render:    runs the function, caches and returns the result.
  On re-renders:      if dependencies unchanged → returns the cached result.
                      if dependencies changed   → re-runs the function,
                                                  caches and returns new result.

  const filtered = useMemo(() => items.filter(...), [items]);
  //    ↑ filtered is an array — same type as what the function returns

  useMemo itself returns nothing extra — just the value your function returned.

Example:
  function ProductList({ products, filterText }) {
    const filtered = useMemo(() => {
      return products.filter(p =>
        p.name.toLowerCase().includes(filterText.toLowerCase())
      );
    }, [products, filterText]);
    // filtered is only recalculated when products or filterText changes

    return filtered.map(p => <p key={p.id}>{p.name}</p>);
  }

  Without useMemo, the filter runs on every render — even renders caused by
  unrelated state changes. With useMemo, it only runs when products or
  filterText actually change.

When to use:
  Expensive computations (sorting or filtering large lists, complex maths)
  that should not run on every render. Also used to stabilize an object
  reference so a child component does not re-render unnecessarily.

When NOT to use:
  Do not wrap every calculation in useMemo. For simple calculations (adding
  two numbers, accessing a property), the overhead of useMemo itself is more
  expensive than just recomputing. Only use it when you have identified a
  real performance problem.

6. useCallback
--------------------------------------------------------------------------------

Technical definition:
  useCallback is a Hook that memoizes a function definition itself (not its
  return value). It accepts a function and a dependency array. It returns the
  same function reference across renders as long as the dependencies have not
  changed. This prevents a new function object from being created on each render.

Simple way:
  useCallback remembers a FUNCTION so that it is not recreated every time the
  component re-renders. This matters when you pass the function as a prop to
  a child component — if the function is "new" on every render, the child
  re-renders unnecessarily even if nothing actually changed.

Syntax:
  const memoizedFn = useCallback(() => doSomething(a), [a]);

Returns:
  The memoized function — the same function reference as long as
  dependencies have not changed.
  The type is exactly the same function type you passed in.

  On first render:    stores the function, returns it.
  On re-renders:      if dependencies unchanged → returns the SAME function
                                                  reference (same object in memory).
                      if dependencies changed   → stores and returns a new function.

  const handleClick = useCallback(() => { ... }, []);
  //    ↑ handleClick is a function — same reference across renders

  The key point: it returns the FUNCTION ITSELF, not the result of calling it.
  To get the result you still have to call handleClick().

Example:
  function Parent() {
    const [count, setCount] = useState(0);

    const handleClick = useCallback(() => {
      console.log("button clicked");
    }, []); // function never changes, no dependencies

    return (
      <div>
        <button onClick={() => setCount(count + 1)}>Count: {count}</button>
        <Child onClick={handleClick} />
      </div>
    );
  }

  const Child = React.memo(function Child({ onClick }) {
    console.log("Child rendered");
    return <button onClick={onClick}>Child Button</button>;
  });

  Without useCallback, handleClick is a new function on every render of Parent.
  Even though nothing in Child changed, Child would re-render because it
  receives a "new" function prop each time. useCallback keeps the same function
  reference, so Child skips re-rendering.

Note — useCallback vs useMemo:
  useMemo    → memoizes the RESULT (a value) of a function
  useCallback → memoizes the FUNCTION itself

  useCallback(() => fn(), []) is equivalent to useMemo(() => () => fn(), [])
  They are the same mechanism, just two different conveniences.

When to use:
  When passing a function as a prop to a child component that is wrapped in
  React.memo, or when a function is a dependency of a useEffect and you do
  not want the effect to re-run on every render.

When NOT to use:
  Same rule as useMemo — do not wrap every function in useCallback. It only
  helps when unnecessary re-renders are an actual problem.

7. useReducer
--------------------------------------------------------------------------------

Technical definition:
  useReducer is a Hook for managing state that involves multiple sub-values or
  where the next state depends on the previous state in complex ways. It accepts
  a reducer function and an initial state, and returns the current state and a
  dispatch function. The reducer is a pure function that takes the current state
  and an action object, and returns the new state.

Simple way:
  useReducer is an alternative to useState for more complex state. Instead of
  calling a setter directly with a new value, you dispatch an "action" that
  describes what happened, and a separate reducer function decides how the state
  changes based on that action. It is the same pattern used by Redux.

Syntax:
  const [state, dispatch] = useReducer(reducerFn, initialState);

Returns:
  An array of exactly two items — a tuple:
  [0] state    → the current state value. Type is whatever your reducer manages.
  [1] dispatch → a function used to send actions to the reducer.
                 Call it with an action object: dispatch({ type: "increment" })
                 dispatch itself returns nothing (void).

  const [state, dispatch] = useReducer(reducer, { count: 0 });
  //     ↑ state object    ↑ dispatch function

  state is read-only — you never mutate it directly.
  dispatch is how you trigger state changes — the reducer decides the new state.

  The reducer function:
  function reducer(state, action) {
    switch (action.type) {
      case "increment": return { count: state.count + 1 };
      case "decrement": return { count: state.count - 1 };
      default: return state;
    }
  }

Example:
  function Counter() {
    const [state, dispatch] = useReducer(reducer, { count: 0 });

    return (
      <div>
        <p>Count: {state.count}</p>
        <button onClick={() => dispatch({ type: "increment" })}>+</button>
        <button onClick={() => dispatch({ type: "decrement" })}>-</button>
      </div>
    );
  }

When to use:
  When a component has multiple related state values that change together, or
  when the update logic is complex enough that it becomes hard to manage with
  several useState calls. Also useful when the next state depends heavily on
  the current state.

When NOT to use:
  Simple, independent state values (a toggle, a counter, a single input field).
  useState is cleaner for those cases.

8. useLayoutEffect
--------------------------------------------------------------------------------

Technical definition:
  useLayoutEffect is identical to useEffect in its API, but it fires
  synchronously after all DOM mutations and before the browser has painted the
  screen. This means it runs before the user sees the update. It blocks the
  browser from painting until it finishes.

Simple way:
  useLayoutEffect works just like useEffect, but it runs earlier — before the
  browser draws the updated page. Use it only when you need to read or change
  something in the DOM before the user sees the result, to avoid a visible
  flicker.

Returns:
  Nothing (void) — same as useEffect.
  The callback inside can optionally return a cleanup function, also void.
  useLayoutEffect itself has no return value you can use.

When to use:
  Measuring a DOM element's size or position and adjusting layout before the
  browser paints. For example, a tooltip that needs to calculate where to
  position itself based on the size of another element.

When NOT to use:
  For everything else, use useEffect. useLayoutEffect can slow down the
  perceived performance of your app because it blocks the paint.

9. useId
--------------------------------------------------------------------------------

Technical definition:
  useId is a Hook that generates a unique, stable ID string for a component
  instance. The ID is consistent between the server and client render, making
  it safe to use for HTML attributes like id and aria-* labels. It was added
  in React 18.

Simple way:
  useId gives you a unique ID that you can attach to form labels and inputs to
  link them properly. The problem it solves is that if you use a hardcoded id
  like "email", having two of the same component on the page creates two HTML
  elements with the same id, which is invalid HTML.

Returns:
  A string — a unique, stable ID for this specific component instance.
  React generates it in a format like ":r0:", ":r1:", ":r2:", etc.
  The same component always gets the same ID across re-renders.
  Two different instances of the same component get different IDs.

  const id = useId();
  //    ↑ a string like ":r3:"

Example:
  function EmailField() {
    const id = useId();
    return (
      <div>
        <label htmlFor={id}>Email</label>
        <input id={id} type="email" />
      </div>
    );
  }

  Two <EmailField /> components on the same page will each get a different id,
  so there is no collision.

When to use:
  Linking form labels to inputs with htmlFor/id pairs, or generating unique
  aria attribute values for accessibility.

10. useTransition
--------------------------------------------------------------------------------

Technical definition:
  useTransition is a Hook introduced in React 18 that lets you mark a state
  update as non-urgent. It returns an isPending boolean and a startTransition
  function. State updates wrapped in startTransition are deferred — React can
  interrupt them to handle more urgent updates (like typing) first.

Simple way:
  useTransition tells React: "this update is not urgent, do it when you have
  time." For example, filtering a large list while the user is typing — the
  typing (urgent) should stay fast, while the list update (non-urgent) can
  wait a moment.

Returns:
  An array of exactly two items — a tuple:
  [0] isPending       → a boolean. True while the non-urgent transition update
                        is still being processed. False when it is done.
                        Use this to show a loading indicator.
  [1] startTransition → a function. Wrap your non-urgent setState call(s) inside
                        it. startTransition itself returns nothing (void).

  const [isPending, startTransition] = useTransition();
  //     ↑ boolean    ↑ function to wrap non-urgent updates

Example:
  function Search() {
    const [query, setQuery] = useState("");
    const [results, setResults] = useState(allItems);
    const [isPending, startTransition] = useTransition();

    function handleChange(e) {
      setQuery(e.target.value); // urgent — updates input immediately

      startTransition(() => {
        setResults(filterItems(e.target.value)); // non-urgent — can wait
      });
    }

    return (
      <div>
        <input value={query} onChange={handleChange} />
        {isPending ? <p>Updating...</p> : results.map(...)}
      </div>
    );
  }

When to use:
  Updating a slow part of the UI (a large filtered list, a tab switch) without
  blocking fast interactions like typing or clicking.
