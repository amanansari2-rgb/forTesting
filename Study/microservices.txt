================================================================================
MICROSERVICES — ARCHITECTURE, PATTERNS, AND RESILIENCE
================================================================================

--------------------------------------------------------------------------------
THINGS YOU SHOULD KNOW FIRST
--------------------------------------------------------------------------------

--- What a Service Is ---

Technical definition:
  A service is an independently runnable unit of software that exposes
  functionality through a well-defined interface (usually an API). It runs
  as its own process, can be deployed separately, and communicates with other
  services or clients over a network.

Simple way:
  A service is a self-contained program that does one job and can be started,
  stopped, and deployed on its own. Other programs talk to it by sending
  requests over the network.


--- What an API Is ---

Technical definition:
  An API (Application Programming Interface) is a defined contract that
  specifies how one piece of software can communicate with another. In the
  context of web services, it typically means a set of HTTP endpoints that
  accept requests and return responses in a structured format (usually JSON).

Simple way:
  An API is a menu. It lists what you can ask for and what you will get back.
  One program sends a request to a URL, and another program reads it and sends
  a response. They do not need to know how each other works internally.


--- What HTTP / REST Is ---

Technical definition:
  HTTP (HyperText Transfer Protocol) is the communication protocol used to
  transfer data over the web. REST (Representational State Transfer) is an
  architectural style for designing APIs over HTTP. It uses standard HTTP
  methods: GET (read), POST (create), PUT (update), DELETE (remove).

Simple way:
  HTTP is the language browsers and services use to talk to each other.
  REST is an agreed set of rules for how to structure those conversations
  so that different systems can understand each other.


--- What Latency and Timeout Are ---

Technical definition:
  Latency is the time delay between sending a request and receiving a response.
  A timeout is a maximum wait duration set by the caller — if a response does
  not arrive within that time, the call is considered failed.

Simple way:
  Latency is how long a request takes to come back. A timeout is the point
  at which you stop waiting and treat it as a failure. In a distributed system
  these matter a lot because network calls can be slow or never respond.


--- What a Distributed System Is ---

Technical definition:
  A distributed system is a collection of independent computing components
  (processes, services, machines) that communicate over a network and appear
  to users as a single coherent system. Distributed systems introduce
  challenges that do not exist in single-process applications — network
  failures, partial failures, data consistency across nodes, and latency.

Simple way:
  A distributed system is multiple separate programs running on different
  machines, working together. The major difference from a single program is
  that the network between them can fail, be slow, or give inconsistent
  results. You have to design for failure.


--- What Fault Tolerance Is ---

Technical definition:
  Fault tolerance is the ability of a system to continue operating correctly
  even when one or more of its components fail. A fault-tolerant system
  degrades gracefully rather than failing completely.

Simple way:
  A fault-tolerant system does not crash just because one part of it has a
  problem. It keeps working, possibly in a reduced capacity, while the broken
  part is failing or recovering.


--- What a Database Schema Is (for CQRS context) ---

Technical definition:
  A database schema is the structure that defines how data is organised in a
  database — the tables, columns, data types, and relationships. A write-
  optimised schema is structured for fast inserts and updates (normalised).
  A read-optimised schema is structured for fast queries (denormalised, flat).

Simple way:
  A database structured for writing data is organised differently from one
  structured for reading it. Writes care about avoiding duplication.
  Reads care about getting all needed data quickly in as few queries as
  possible.

--------------------------------------------------------------------------------
1. MICROSERVICE
--------------------------------------------------------------------------------

Technical definition:
  A microservice is a small, independently deployable service that implements
  a single, well-defined business capability. A microservices architecture
  structures an application as a collection of such services, each running
  in its own process, owning its own data store, and communicating with others
  via lightweight protocols (HTTP/REST, gRPC, or message queues). Each service
  can be developed, deployed, scaled, and updated independently.

Simple way:
  Instead of building one large program that does everything, you build many
  small programs — each responsible for exactly one job. A shopping app might
  have separate services for users, products, orders, payments, and
  notifications. Each one runs on its own, talks to the others through APIs,
  and can be updated without touching the rest.

Core characteristics:
  - Single responsibility — each service does one business thing
  - Independent deployment — deploy one service without redeploying others
  - Owns its own data — each service has its own database, not a shared one
  - Communicates over the network — via HTTP, gRPC, or messages
  - Can use different technologies — one service in C#, another in Python

Example:
  An e-commerce platform split into microservices:

  [ Order Service ]   → manages creating and tracking orders
  [ Product Service ] → manages product catalogue and inventory
  [ User Service ]    → manages accounts and authentication
  [ Payment Service ] → handles payment processing
  [ Email Service ]   → sends notification emails

  Each one is a separate program. The Order Service calls the Payment Service
  over HTTP to charge a customer. They are independent — the Product Service
  can be updated without touching the Order Service.

--------------------------------------------------------------------------------
1a. TYPES OF MICROSERVICES
--------------------------------------------------------------------------------

Microservices can be categorised in two ways — by the role they play in the
system, and by how they communicate with other services.


--- By Role ---

  Domain Service (Business Service):

  Technical definition:
    A Domain Service implements a specific, bounded business capability.
    It owns its own data and encapsulates all business logic for that domain.
    These are the core building blocks of any microservices system.

  Simple way:
    These are the main services that do the actual business work.
    Each one maps to one thing the business does.

  Examples: Order Service, Payment Service, Product Service, User Service.


  Infrastructure / Utility Service:

  Technical definition:
    An Infrastructure Service handles a technical cross-cutting concern that
    is not specific to any one business domain. It provides shared technical
    functionality that multiple domain services depend on.

  Simple way:
    These are the support services — not business logic, but technical
    plumbing that every other service needs.

  Examples: Authentication Service, Email/Notification Service,
            Logging Service, Configuration Service.


  Aggregator Service:

  Technical definition:
    An Aggregator Service calls multiple downstream services, collects their
    responses, and combines them into a single unified response for the
    caller. It reduces the number of round trips a client has to make.

  Simple way:
    Instead of a client making five separate calls to five services, the
    aggregator makes those five calls internally and returns one combined
    answer. The client makes one call instead of five.

  Example:
    A Dashboard Service calls the User Service, Order Service, and
    Recommendation Service, and returns all combined data in one response.


  BFF — Backend for Frontend:

  Technical definition:
    A BFF is a specialised variant of the Aggregator pattern where a
    separate backend service is built for each type of client (mobile app,
    web browser, desktop app). Each BFF is tailored to return exactly the
    data shape and volume its specific client needs, rather than a one-size-
    fits-all response.

  Simple way:
    A mobile app and a web app often need different versions of the same
    data — the mobile app wants compact, minimal data to save bandwidth,
    the web app wants full detailed data. Instead of one API that compromises
    on both, you build a separate thin backend for each client type.

  Example:
    Mobile BFF  → returns compact order summary (id, status, total only)
    Web BFF     → returns full order details (items, history, shipping info)
    Both call the same Order Service behind the scenes.


--- By Communication Style ---

  Synchronous Communication (Request / Response):

  Technical definition:
    The calling service sends a request and waits (blocks) until it receives
    a response. The caller is dependent on the called service being available
    at that moment. Common protocols: HTTP/REST and gRPC.

  Simple way:
    You ask a question and wait for an answer before doing anything else.
    Fast and simple, but if the service you are calling is slow or down,
    your service is also blocked and slow.

  When to use:
    When you need an immediate answer to continue — e.g., calling a payment
    service and needing a success/failure before confirming an order.


  Asynchronous Communication (Message / Event-Driven):

  Technical definition:
    The calling service sends a message to a message broker (queue or event
    stream) and continues without waiting for a response. The receiving
    service processes the message when it is ready. Common tools: RabbitMQ,
    Azure Service Bus, Apache Kafka.

  Simple way:
    You drop a note in a mailbox and walk away. The other service reads the
    note and acts on it in its own time. Your service does not wait around.
    The two services are decoupled — they do not need to be running at the
    same time.

  When to use:
    When you do not need an immediate response — e.g., after placing an
    order, send an event to the Email Service to notify the customer.
    The order is confirmed immediately; the email sends in the background.

  Trade-off:
    Synchronous is simpler but creates tight coupling.
    Asynchronous is more resilient but harder to trace and debug.

1b. STATEFUL vs STATELESS SERVICES
--------------------------------------------------------------------------------

--- Stateless Service ---

Technical definition:
  A stateless service does not retain any data about previous requests
  between calls. Each request is entirely self-contained — it includes all
  the information the service needs to process it. The service has no memory
  of past interactions. Because no request depends on a specific instance,
  any running instance of the service can handle any incoming request.

Simple way:
  A stateless service has no memory between requests. Every request arrives
  as if it is the first one. The service reads the request, does its job,
  returns the result, and forgets everything.

  Like a calculator — you press buttons, it gives an answer. It does not
  remember what you calculated last time.

Why this matters for scaling:
  Because no instance holds any client-specific memory, you can run as many
  copies (instances) of the service as you need. Any instance handles any
  request. Adding more instances to handle more traffic is straightforward.

Example:
  A Tax Calculation Service. You send: product price + region code.
  It returns the tax amount. No memory needed between calls.


--- Stateful Service ---

Technical definition:
  A stateful service retains data (state) between requests from the same
  client or session. The outcome of a request may depend on previous
  interactions. If state is stored in the service's memory (in-process),
  the same instance must handle all requests from that client — called a
  "sticky session." This makes horizontal scaling complex.

Simple way:
  A stateful service remembers things between requests. The result of your
  current request can depend on what happened in previous ones.

  Like a conversation — the second message only makes sense in the context
  of the first.

Why this creates problems in microservices:
  If the state lives inside one service instance's memory and that instance
  crashes or restarts, the state is lost. Also, if you scale up and run
  three instances, you must ensure each client always reaches the same
  instance — which complicates load balancing.

Example:
  A multi-step checkout wizard where each step depends on the previous step's
  data being remembered.


--- The Recommended Approach: Externalise State ---

Technical definition:
  Instead of holding state in the service's own memory, the service stores
  and retrieves state from an external store — a database or a distributed
  cache (e.g., Redis). The service code itself remains stateless. Any instance
  can serve any request because all instances share the same external state
  store.

Simple way:
  Make the service code stateless by moving the memory outside the service.
  The service does not remember anything — instead, it reads and writes to
  a shared database or cache. Every instance of the service can do this, so
  any instance can handle any request.

  Stateful service (bad for scaling):
    Client → always must hit Instance 1 (which holds the session in memory)

  Stateless service + external state store (scalable):
    Client → Instance 1 or 2 or 3 (all read from the same Redis/database)

In microservices, the standard rule is:
  Keep services stateless. Externalise state to a database or cache.

2. MONOLITHIC vs MICROSERVICE
--------------------------------------------------------------------------------

--- Monolithic Architecture ---

Technical definition:
  A monolithic application is one in which all components — UI, business logic,
  data access — are built and deployed as a single, unified unit. All modules
  run in the same process and share the same database. Scaling requires
  replicating the entire application, and a failure in any one module can
  potentially bring down the whole system.

Simple way:
  A monolith is one big program that does everything. All the code lives
  together, compiles together, and deploys together. If you want to change
  the payment logic, you redeploy the entire application.

  Think of it as a single large building where everything — offices, storage,
  kitchen — is in one structure. To expand the kitchen, you have to work on
  the whole building.


--- Microservice Architecture ---

Technical definition:
  A microservices architecture decomposes an application into a collection of
  small services, each independently deployable, each owning its own data, and
  each communicating over a network. Services are loosely coupled — a change
  in one does not require changes in others as long as the API contract is
  maintained.

Simple way:
  Microservices is many small buildings (services), each doing one job.
  To expand the kitchen (payment service), you only work on that building —
  everything else keeps running untouched.


--- Side-by-Side Comparison ---

  Factor               | Monolithic                    | Microservices
  ---------------------|-------------------------------|--------------------------------
  Deployment           | Whole app redeploys together  | Each service deploys separately
  Scaling              | Scale the entire app          | Scale only the service you need
  A bug in one area    | Can crash the whole app       | Only that service is affected
  Technology choice    | One language/framework        | Each service can use different tech
  Development speed    | Fast to start (less setup)    | Slower to start (more infrastructure)
  Testing              | Easier end-to-end testing     | Harder — need to test across services
  Network calls        | No network — in-process calls | Network calls between services
  Data management      | One shared database           | Each service owns its own database
  Team size            | Works well for small teams    | Works well for large teams (per service)

When to use Monolith:
  Small teams, early-stage products, or applications where the complexity of
  distributed systems outweighs the benefits. Starting as a monolith and
  splitting later is a valid and common approach.

When to use Microservices:
  Large applications with multiple teams, services that need to scale
  independently, or when different parts of the system have very different
  technology requirements.

Trade-off to be honest about:
  Microservices solve some problems but introduce others — network latency,
  distributed data consistency, more infrastructure to manage, and harder
  debugging. They are not automatically better. They are a trade-off.


--- Greenfield vs Brownfield ---

  These two terms describe the context in which you build a microservices
  system — whether you are starting from scratch or migrating an existing one.

  Greenfield:

  Technical definition:
    A Greenfield project is one built from scratch with no pre-existing system
    or legacy constraints. The team designs and implements the microservices
    architecture from the beginning without needing to integrate with or
    replace any existing application.

  Simple way:
    You start with a blank slate. No old system to worry about. You design
    everything the right way from day one.

  Reality: Greenfield is less common. Most real-world work involves existing
  systems that have to keep running while being changed.


  Brownfield:

  Technical definition:
    A Brownfield project involves migrating or evolving an existing system —
    typically a monolith — into a microservices architecture. The existing
    system is live and in production, so the migration must be done gradually
    without disrupting running users or business operations.

  Simple way:
    You have an existing monolith that is already live and serving users.
    You cannot simply stop it and rewrite everything from scratch. You have
    to carefully extract pieces from it while it keeps running — like doing
    surgery on a patient who is awake.

  The main challenge:
    The old monolith has years of tightly coupled business logic, shared
    database tables, and interdependencies. Untangling this safely takes
    time and careful planning.


  Strangler Fig Pattern (the standard Brownfield approach):

  Technical definition:
    The Strangler Fig Pattern is a migration strategy where new functionality
    or extracted capabilities are implemented as new microservices, and traffic
    for those capabilities is gradually redirected from the monolith to the
    new services. Over time the monolith handles less and less until it is
    fully replaced or decommissioned. Named after the strangler fig tree, which
    grows around an existing tree and eventually replaces it.

  Simple way:
    You do not rewrite everything at once. You pick one piece of the monolith,
    build a new microservice for it, point traffic to the new service, and
    leave everything else untouched. Then repeat for the next piece.
    The monolith slowly shrinks. The new services grow. Eventually the monolith
    is gone.

  Step-by-step:
    1. Identify one module in the monolith to extract (e.g., User Management)
    2. Build a new standalone User Service with the same functionality
    3. Route all /users traffic to the new service (via API Gateway)
    4. The monolith still handles all other traffic unchanged
    5. Decommission the User module from the monolith
    6. Repeat for the next module (e.g., Orders)

  Anti-Corruption Layer (ACL):

  Technical definition:
    An Anti-Corruption Layer is a translation layer placed between a new
    microservice and the legacy monolith (or old database) it must temporarily
    communicate with. It prevents the old system's data model and terminology
    from polluting the design of the new service. The new service uses its own
    clean domain model; the ACL translates between the two.

  Simple way:
    The old monolith uses its own naming and data structure. The new service
    uses a clean, modern design. Rather than forcing the new service to speak
    the old system's language, you put a translator in between. The new service
    stays clean; the translator handles the messy conversion.

3. API GATEWAY
--------------------------------------------------------------------------------

Technical definition:
  An API Gateway is a server that acts as the single entry point for all
  client requests in a microservices architecture. It receives requests from
  external clients, routes them to the appropriate downstream microservice(s),
  and returns the response. It can additionally handle cross-cutting concerns
  such as authentication, rate limiting, SSL termination, request/response
  transformation, logging, and load balancing — centralising these so
  individual services do not have to implement them.

Simple way:
  An API Gateway is the front door to your entire system. Clients (mobile app,
  browser, external partner) always knock on one door. The gateway decides
  which service should handle the request and forwards it there. The client
  never talks to individual services directly — it does not even know they
  exist.

Without an API Gateway:
  A client would need to know the address of every microservice, call each
  one separately, and handle authentication and errors for each individually.
  This tightly couples the client to your internal architecture.

With an API Gateway:
  The client knows one address. The gateway handles routing, security, and
  aggregation. Your internal services can change, move, or be split without
  affecting the client.

What an API Gateway typically handles:

  Routing:
    Receives a request for /orders and routes it to the Order Service.
    Receives /products and routes to the Product Service.

  Authentication / Authorisation:
    Validates a JWT token or API key once at the gateway — services behind
    it do not all need to repeat this validation.

  Rate Limiting:
    Limits how many requests a single client can make per second/minute to
    protect services from overload.

  SSL Termination:
    Handles HTTPS at the gateway level so internal services can communicate
    over plain HTTP in a trusted network.

  Load Balancing:
    If a service has multiple instances running, the gateway distributes
    incoming requests across them.

  Request Aggregation:
    A single client request may require data from multiple services. The
    gateway can call several services and combine the responses into one
    before returning to the client.

Example flow:
  Client → GET /api/orders/123
  ↓
  API Gateway receives request
  → validates auth token
  → routes to Order Service at http://order-service/orders/123
  → Order Service responds with order data
  ↓
  API Gateway returns response to client

  Client only ever sees the gateway's address. Order Service address is
  internal and hidden.

Common implementations:
  AWS API Gateway, Azure API Management, Kong, Ocelot (.NET), Nginx, Traefik.

4. CQRS — COMMAND QUERY RESPONSIBILITY SEGREGATION
--------------------------------------------------------------------------------

Technical definition:
  CQRS is an architectural pattern that separates the model for handling write
  operations (Commands) from the model for handling read operations (Queries).
  A Command changes state and returns no data (or just a success/failure
  confirmation). A Query reads state and returns data but makes no changes.
  By separating these, each side can be independently optimised, scaled, and
  structured for its specific workload.

Simple way:
  In a standard design, one model handles both reading and writing data.
  CQRS splits this into two separate models:
  - The write side (Command) handles changes — create, update, delete.
  - The read side (Query) handles data retrieval.

  The reason is that reading and writing often have completely different
  requirements. Reads are usually far more frequent and need fast, flat
  data. Writes need to enforce business rules and data integrity.


--- Command ---

Technical definition:
  A Command is an instruction that expresses the intent to change state.
  It is named in imperative form (CreateOrder, CancelPayment, UpdateAddress).
  A command is processed by a Command Handler, which validates and executes
  the business logic, then writes to the write data store. Commands do not
  return data — only a success or failure indication.

Simple way:
  A command is a "do this" instruction. It changes something.
  Examples: "Place this order", "Cancel this booking", "Change this address."
  After it runs, you know it succeeded or failed — but it does not return
  the resulting data to you.


--- Query ---

Technical definition:
  A Query is a request to read data without producing any side effects.
  It is processed by a Query Handler, which reads from the read data store
  and returns a result. Because queries are side-effect-free, they can be
  freely cached and replicated.

Simple way:
  A query is a "give me this data" request. It reads something and returns it.
  It changes nothing. Examples: "Get order #123", "List all products."


--- Why Separate Read and Write Stores ---

  In a traditional design, one database serves both reads and writes. This
  is simple but has a cost: the database schema must satisfy both workloads
  simultaneously, which often means it is not optimal for either.

  Write store (optimised for writes):
    Normalised schema — data split across multiple related tables to avoid
    duplication. Fast for inserts and updates. Enforces data integrity.

  Read store (optimised for reads):
    Denormalised schema — data pre-joined and flattened into views or
    separate read tables. A single query returns everything needed without
    joining multiple tables. Fast for reads.

  The read store is kept in sync with the write store, typically through
  events — when the write side changes data, it raises an event, and the
  read side updates its own store accordingly.

Example:
  Write side: An order is placed → validated → written to the Orders table
              and OrderItems table (normalised, multiple tables).

  Read side: The order data is pre-combined into a flat OrderSummary read
             model, ready to be returned instantly for a dashboard view.

When to use CQRS:
  - When read and write workloads have very different performance requirements
  - When the read side needs a very different data shape than the write side
  - In complex domains where write logic (business rules) should be strictly
    separated from read logic (data presentation)

When NOT to use CQRS:
  Simple CRUD applications (Create, Read, Update, Delete) with no complex
  business logic get no benefit from CQRS and only get added complexity.
  Do not apply it to every service by default.

5. SERVICE DISCOVERY
--------------------------------------------------------------------------------

Technical definition:
  Service Discovery is the mechanism by which services in a microservices
  architecture locate the network addresses (IP and port) of other services
  at runtime. Because services in modern deployments (containers, cloud) are
  assigned dynamic IP addresses that change on restart or scaling events,
  hardcoded addresses are not viable. A Service Registry maintains an up-to-date
  record of all running service instances and their addresses.

Simple way:
  In a microservices system, services need to call each other. You cannot
  hardcode addresses because they change — especially when services are
  containerised (Docker/Kubernetes) and restart on different machines with
  new IP addresses.

  Service Discovery solves this: instead of knowing an address, a service
  asks a central registry "where is the Order Service right now?" and gets
  the current address back.


--- Service Registry ---

Technical definition:
  A Service Registry is a database that stores the location (host and port)
  of all available service instances. Services register themselves when they
  start (self-registration) and deregister when they shut down. The registry
  performs health checks to remove instances that are no longer responding.

Simple way:
  The registry is the phone book of your microservices system. When a service
  starts, it adds its address to the book. When it stops, it removes itself.
  Other services look up the book to find who they need to call.

Common registries:
  Consul, Netflix Eureka, etcd, Apache ZooKeeper. Kubernetes has its own
  built-in service discovery.


--- Two Types of Service Discovery ---

  Client-Side Discovery:

  Technical definition:
    The calling service (client) queries the Service Registry directly to
    get the list of available instances of the target service, then applies
    a load-balancing algorithm (e.g., round robin) to choose one and calls
    it directly.

  Simple way:
    The caller does the lookup itself. It asks the registry, gets a list of
    available addresses, picks one, and calls it.

  Diagram:
    Service A → asks Registry → gets list of Service B instances
    Service A → picks one → calls Service B directly


  Server-Side Discovery:

  Technical definition:
    The calling service sends its request to a load balancer or router, which
    queries the Service Registry on the caller's behalf, selects an instance,
    and forwards the request. The caller does not interact with the registry
    directly.

  Simple way:
    The caller sends the request to a router/load balancer and lets it handle  
    the lookup and forwarding. The caller does not need to know about the
    registry at all.

  Diagram:
    Service A → calls Load Balancer → Load Balancer asks Registry
    → Load Balancer forwards to Service B instance


  Which is more common:
    Kubernetes uses server-side discovery — services call a stable DNS name
    and Kubernetes handles routing to the correct instance internally.

6. RETRY PATTERN AND CIRCUIT BREAKER
--------------------------------------------------------------------------------

--- Why These Patterns Exist ---

  In a distributed system, network calls between services will sometimes fail
  — not because the service is permanently broken, but due to transient
  (short-lived) issues: a brief network hiccup, a service momentarily busy,
  a temporary timeout.

  Without any resilience strategy, a single failed call causes a visible
  error to the user, even if retrying immediately would have succeeded.

  But if a service is genuinely down for minutes, blindly retrying thousands
  of times wastes resources, keeps connections open, and can make the
  failing service's recovery even slower.

  These two patterns address these two different failure scenarios:
  - Retry: for transient (short-lived) failures
  - Circuit Breaker: for sustained (prolonged) failures


--- Retry Pattern ---

Technical definition:
  The Retry pattern automatically re-executes a failed operation a defined
  number of times before reporting failure. It is used for transient faults
  that are likely to resolve on their own within a short period. Retries should
  be combined with a delay strategy (exponential backoff with jitter) to avoid
  overloading a struggling service with repeated immediate requests.

Simple way:
  If a request fails, try again. If it fails again, try one more time. After
  a set number of attempts (e.g., 3), give up and report a failure. This
  handles the case where a network blip or momentary overload caused the
  first attempt to fail, but the second attempt succeeds.


--- Exponential Backoff ---

Technical definition:
  Exponential backoff is a delay strategy where the wait time between retries
  increases exponentially with each attempt. Adding jitter (a random additional
  delay) avoids the thundering herd problem — where many services retry at
  exactly the same moment and simultaneously overload a recovering service.

Simple way:
  Instead of retrying immediately (which hammers a struggling service), you
  wait longer after each failure:
  - Attempt 1 fails → wait 1 second → retry
  - Attempt 2 fails → wait 2 seconds → retry
  - Attempt 3 fails → wait 4 seconds → retry
  - Give up

  Jitter adds a small random amount to each wait to spread retries out.


--- What Should NOT Be Retried ---

  Not all errors are transient. Retrying a permanent error wastes time.

  Retry these   → 408 Request Timeout, 429 Too Many Requests, 503 Service
                  Unavailable, 504 Gateway Timeout, network timeouts
  Do NOT retry  → 400 Bad Request (your request is wrong — retrying will
                  always fail), 401 Unauthorized, 403 Forbidden, 404 Not Found


--- Circuit Breaker ---

Technical definition:
  The Circuit Breaker pattern wraps calls to a remote service and monitors
  for failures. When the number of consecutive failures exceeds a defined
  threshold, the circuit "opens" — subsequent calls immediately return a
  failure response without actually attempting to call the service. After a
  configured timeout, the circuit moves to a "half-open" state, allowing a
  limited number of test requests through. If those succeed, the circuit
  closes and normal operation resumes. If they fail, the circuit opens again.

Simple way:
  Named after the electrical circuit breaker in a fuse box — which cuts
  power when it detects a dangerous overload rather than letting a fault
  cause more damage.

  In software: if a downstream service keeps failing, stop trying to call it.
  Instead of every request waiting for a timeout and then failing, fail
  immediately and fast. This protects your service from slow responses
  piling up and protects the failing service from being bombarded while it
  recovers.


--- The Three States ---

  CLOSED (normal):
    All requests go through to the service. Failures are counted.
    The circuit stays closed while failures stay below the threshold.

    Service A ──── request ────► Service B   ✓

  OPEN (failure threshold exceeded):
    Requests are immediately rejected — the call to Service B is not made.
    A timer starts. This is the "fail fast" state.
    The caller receives an error instantly, not after a long timeout.

    Service A ──X  (circuit open — no call made)

  HALF-OPEN (testing recovery):
    After the timer expires, the circuit allows a small number of test
    requests through to check if Service B has recovered.
    - If they succeed → circuit returns to CLOSED
    - If they fail   → circuit returns to OPEN, timer resets

    Service A ──── 1 test request ────► Service B
                                          ↓
                   success → CLOSED / failure → OPEN


--- Retry + Circuit Breaker Together ---

  They are complementary, not alternatives.

  Retry alone:
    Works for occasional transient failures. But if a service is down for
    5 minutes, every call retries 3 times — that is 3x the traffic hitting
    an already-struggling service.

  Circuit Breaker alone:
    Stops calls once a threshold is hit. But without retry, a single failed
    request is immediately reported as a failure even if it would have
    succeeded on the second attempt.

  Together:
    Retry handles the brief hiccups. Circuit Breaker handles the prolonged
    outage. The retry runs first. If retries repeatedly fail and the circuit
    breaker threshold is hit, the circuit opens and stops further retries
    until recovery.

  Typical combined policy (Polly — the .NET resilience library):
    - Retry up to 3 times with exponential backoff
    - Circuit Breaker opens if 5 consecutive failures occur
    - Circuit stays open for 30 seconds
    - Half-open: allow 1 request to test recovery

  Microservice         | Small, independently deployable service for one business
                       | capability, owning its own data

  Async Communication  | Caller sends and continues — message queue/event stream
  Stateless Service    | No memory between requests — any instance handles any call
  Stateful Service     | Remembers between requests — harder to scale
  Externalise State    | Keep service code stateless, store state in DB or Redis
  Greenfield           | Building microservices from scratch — no legacy system
  Brownfield           | Migrating an existing system (monolith) to microservices
  Strangler Fig        | Gradual migration — extract one piece at a time until
                       | the monolith is fully replaced
  Anti-Corruption Layer| Translator between new service and legacy system models
  Monolith             | One large application — everything compiled and deployed
                       | together as a single unit
  API Gateway          | Single entry point for all clients — handles routing,
                       | auth, rate limiting, aggregation
  CQRS                 | Separate models for write (Command) and read (Query)
                       | operations — each optimised for its workload
  Service Registry     | Central phone book of all running service addresses
  Service Discovery    | Mechanism for services to find each other dynamically
                       | at runtime — client-side or server-side
  Retry Pattern        | Automatically re-attempt a failed call for transient
                       | faults — with backoff between attempts
  Exponential Backoff  | Increasing wait time between retries to reduce pressure
                       | on a struggling service
  Circuit Breaker      | Stops calling a failing service when a failure threshold
                       | is hit — fail fast, allow recovery time
  Circuit States       | Closed (normal) → Open (fail fast) → Half-Open (test)
